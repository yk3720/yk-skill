# Techmap

実務経験を持つ理系エンジニアが「今日初めてこの技術を調べた」状態から本質を掴めるHTMLを生成し、surge.shに公開する。

## 対象読者

- **実務経験のある理系エンジニア**（数学・物理・電気・情報などの基礎素養は前提）
- 対象トピックは未経験でも、**構造的な説明があればすぐ理解できる**タイプ
- 一つを調べると関連知識も気になる好奇心旺盛な学習者

> **前提として持っている知識**: 微分積分・線形代数・確率統計・電気回路・プログラミング基礎など、理系学部卒以上の素養。これらに関わる用語は説明不要。

## ゴール

> 本質的な概念を既存の知識と対応させて理解し、自分の言葉で説明できる。さらに「次に何を学べばいいか」が分かり、学習の連鎖が起きる状態

---

## ワークフロー

### Phase 0: リサーチ（サブエージェント）

**exploreサブエージェント**を起動して対象トピックを徹底調査する。

```
Task({
  subagent_type: "explore",
  description: "技術トピックの徹底調査",
  prompt: `
    以下の技術について徹底的に調査してください：
    【対象】{トピック名}

    調査項目：
    1. この技術がなぜ生まれたのか（背景・歴史・解決したかった課題）
    2. この技術は何を解決するものか（Why・存在意義）
    3. 主要な概念・用語の一覧（初学者がまず知るべきもの）
    4. 典型的なユースケース・活用場面（エンジニアの実務・研究に関連する文脈を優先）
    5. 初学者がはまりやすい落とし穴・よくある誤解（実際にハマった例があれば具体的に）
    6. 料金プラン・ライセンス体系（ツール・サービスの場合。無料枠・有料プランの違いを含む）
    7. 関連する公式ドキュメント・参考サイトのURL（できるだけ複数）
    8. この技術と関連する周辺技術・発展的なトピック

    結果は箇条書きで整理して返してください。
  `
})
```

### Phase 1: ソース読み込み

1. URLなら `WebFetch` で取得、ドキュメントなら内容を分析
2. **情報の全体像を把握**（漏れなく）
3. Phase 0のリサーチ結果と照合して抜け漏れを確認
4. 「実務経験があるエンジニアが初めてこの技術を見た場合」の視点で整理する

### Phase 2: 用語対応表を作成

新しい技術用語を、読者がすでに知っている概念・構造と対応させて説明する。  
「やさしく言い換える」のではなく、**既知の知識体系に接続する**。

**略称・英語表記のルール**: 英語の略称や頭字語は、**初出時に正式名称と日本語訳を併記**する。  
（例：API → Application Programming Interface（アプリケーション プログラミング インタフェース）、ORM → Object-Relational Mapping（オブジェクト関係マッピング））

| 技術用語 | 正式名称（略称の場合） | 一言の定義 | 既知の概念との対応 |
|---------|---------------------|-----------|------------------|
| （例）ORM | Object-Relational Mapping | DBのテーブルをオブジェクトとして扱う変換層 | 型付き構造体とテーブルスキーマの写像関係。行列の基底変換に近い発想 |

**詳細** → [references/term-dictionary.md](references/term-dictionary.md)

### Phase 3: たとえ話を3つ以上考える

**たとえ話は「新人エンジニアでも直感的に理解できること」を最優先とする。**  
専門知識を前提にした比喩より、概念の核心を掴める身近・具体的なたとえを積極的に選ぶ。  
ただし正確性は損なわないこと。たとえが示す構造と実際の技術構造が一致しているかを確認する。

**良いたとえ話の条件**:
- 「なぜそう動くのか」の直感が1〜2文で伝わる
- 新人エンジニアが「ああ、そういうことか」と声を出せるくらい具体的
- 技術的に誤解を生まない（正確性を犠牲にしない）

| 種類 | 例 |
|-----|-----|
| 身近な仕組み・日常 | 宅配便の仕分けセンター、図書館の貸出管理、工場の組み立てライン、レストランのオーダーシステム |
| ソフトウェア工学（入門レベル） | 関数の呼び出しスタック、キャッシュとメモ帳、ログとレシート |
| 数学・アルゴリズム（補足として） | グラフ理論、状態機械、収束条件、計算量のオーダー |

### Phase 4: 情報を絞り込む

- **「まず理解する3つ」**を決定（これだけ分かれば入門完了レベル）
- 全体の中の位置づけを示す（マップ感覚）
- 詳細・発展的内容は後半に配置

### Phase 5: 論調・スタイルを設計

**「冷静で誠実な技術者」**の語り口で書く。

- 煽らない、大げさにしない
- 読者の理解力を信頼し、**説明を過度に簡略化しない**（上から目線の簡略化は逆効果）
- 正確さを最優先にしつつ、構造・背景・理由をセットで説明する
- **専門用語は積極的に使う**。ただし対象技術固有の用語は、初出時に必ず定義を添える
- **英語の略称・頭字語は初出時に正式名称と日本語訳を展開する**（例：`TCP（Transmission Control Protocol：伝送制御プロトコル）`）
- キャラクター対話なし（ナレーション形式）
- 疑問→回答の構造で進める（見出し疑問文 → 本文で答える形）

### Phase 6: HTML生成

1. Tailwind CSS CDN使用（絵文字禁止・Lucide Iconsを使用）
2. 用語解説ボックスを初出時に配置（英語略称は正式名称と日本語訳を記載。例：`HTML = HyperText Markup Language（ハイパーテキスト マークアップ言語）`）
3. コード例の前に「このコードがやること」を日本語で説明
4. **PDFボタン**を右下に固定配置（`window.print()` で実装。`@media print` でボタン・目次を非表示にし、印刷時に各セクションが途中で切れないよう `break-inside: avoid` を適用）

**セクション構成（この順番で配置すること）**:

| 順 | セクション | 内容 |
|---|-----------|------|
| 1 | ひとことで言えば | 一文サマリー・対象読者・この技術がカバーする範囲 |
| 2 | なぜ生まれたのか | 歴史的背景・解決したかった課題・登場の経緯 |
| 3 | まず理解する3つ | 入門完了に必要な核心概念を3つに絞る |
| 4 | 主要概念・アーキテクチャ | 詳細な概念説明・図・コード例（たとえ話を3つ以上含む） |
| 5 | 初心者がはまりやすい落とし穴 | 実際によくある誤解・失敗パターンと回避策 |
| 6 | 料金プラン *(ツール・サービスの場合のみ)* | 無料枠・有料プランの違い・コスト感の目安 |
| 7 | 専門用語集 | 5〜15語、定義付き（詳細は下記） |
| 8 | 次に学ぶと良いもの | 関連トピック一覧 |
| 9 | 参考文献・URL | クリッカブルなリンク一覧 |

**専門用語集セクションの構成**:
- このトピックを理解・活用するうえで**覚えておくべき専門用語**を選定してまとめる
- 本文中に登場した用語に限らず、**関連する重要語も含める**
- 各用語は以下の形式で記載する:
  - 用語名（英語略称の場合は正式名称と日本語訳を併記）
  - 一文の定義
  - 関連する他の用語へのクロス参照（あれば）
- 用語数の目安: 5〜15語（多すぎると参照性が下がる）

**詳細** → [references/html-structure.md](references/html-structure.md)

### Phase 7: 批判的レビュー（サブエージェント）

**実務経験のある理系エンジニアが初めてこの技術を学ぶ**視点で、忖度なくレビューする。

```
Task({
  subagent_type: "generalPurpose",
  description: "経験者エンジニア視点での図解レビュー",
  readonly: true,
  prompt: `
    あなたは「数学・物理・情報系の素養があり実務経験も十分あるエンジニアだが、{トピック名}は今日初めて調べた」という立場です。
    理解力は高く、論理的な説明があればすぐ掴めますが、過度に簡略化された説明や小学生向けの比喩には違和感を覚えます。

    以下のHTML図解を読んで、**忖度なく**批判してください：

    【HTMLファイルパス】{path}

    ## レビュー観点

    1. **わからない言葉**: 対象技術固有の用語で説明なしに出てくるものはないか。英語の略称で正式名称・日本語訳が展開されていないものはないか
    2. **たとえ話の適切さ**: 新人エンジニアが「なるほど」と思えるたとえか。「何と何が対応しているのか」が明確か。技術的に誤った構造になっていないか
    3. **情報の過不足**: 「もっと欲しかった情報」「多すぎて辛かった情報」はないか
    4. **生まれた背景**: なぜこの技術が必要とされたのか、歴史・文脈が伝わるか
    5. **落とし穴の有用性**: 初心者が実際にハマりそうな点が具体的に書かれているか
    6. **料金プランの明確さ** *(ツールの場合)*: 無料枠の制限・有料プランの価格感が分かるか
    7. **学習の連鎖**: 読み終わった後、「次にこれを調べたい」と思えたか
    8. **参考文献の質**: URLや参考資料はすぐ確認できるか

    ## 出力形式

    ### 致命的な問題（これがあると理解できない）
    - ...

    ### 改善すべき点（直せばもっと伝わる）
    - ...

    ### 良かった点（このまま残すべき）
    - ...

    ### 総評
    5段階評価: ⭐⭐⭐☆☆（3/5）のように
    一言: 「○○だから△△な人には難しい」のように
  `
})
```

### Phase 8: ブラッシュアップ

Phase 7のレビュー結果を踏まえてHTMLを修正する。

**優先度**:
1. 致命的な問題 → 必ず修正
2. 改善すべき点 → 可能な限り対応
3. 良かった点 → 維持

**よくある修正パターン**:
- 用語解説ボックスの追加
- 過度に簡略化されたたとえ話を、技術的・学術的に正確なたとえに差し替え
- 参考URLの追加・整理
- 関連トピックの追加

### Phase 9: デプロイ

1. `output/` ディレクトリに `{スラッグ}.html` として保存
2. デプロイ実行:

**Windows（PowerShell）の場合:**

```powershell
npx --yes surge output/{スラッグ}.html --domain techmap-[スラッグ].surge.sh
```

3. 完了報告:

```
完成・公開完了: 【図解タイトル】

（図解の内容を1〜2文で要約）

公開URL:
https://techmap-スラッグ.surge.sh

この図解で理解できること:
- （主要ポイントを3〜5個）

次に学ぶと良いトピック:
- （関連トピックを2〜3個）
```

## 品質チェックリスト

### 必須

- [ ] 冒頭に「ひとことで言えば」のサマリーがある
- [ ] **なぜ生まれたのか**セクションがある（歴史・背景・解決した課題）
- [ ] 「まず理解する3つ」のような絞り込みがある
- [ ] 対象技術固有の用語には「用語解説ボックス」がある（初出時）
- [ ] 英語略称・頭字語は初出時に正式名称と日本語訳が展開されている（例：`API（Application Programming Interface：アプリケーション プログラミング インタフェース）`）
- [ ] たとえ話が3つ以上ある（新人エンジニアでも直感的に「なるほど」と思えるもの。かつ技術的に誤解を生まないもの）
- [ ] コード例の前に「このコードがやること」がある（日本語）
- [ ] **初心者がはまりやすい落とし穴**セクションがある（具体的な失敗パターンと回避策）
- [ ] **料金プラン**セクションがある（ツール・サービスの場合のみ）
- [ ] **専門用語集**（5〜15語、定義付き）が「次に学ぶと良いもの」の直前にある
- [ ] **関連トピック**（次に学ぶと良いもの）がある
- [ ] **参考文献・URL**がクリッカブルなリンクで末尾にある

### デザイン

- [ ] Lucide iconのみ使用（絵文字禁止）
- [ ] フローチャートまたは概念図が含まれている
- [ ] スマホでも読みやすいレスポンシブ対応
- [ ] 目次がフローティング表示される（デスクトップ）
- [ ] **「PDFで保存」ボタン**が右下に固定表示されている（印刷時は自動非表示）
