Cursor Agents Window

エージェント中心開発の実践ガイド

対象: 実務経験のある理系エンジニア ｜ Cursor 3（2026年4月2日）リリース機能

ひとことで言えば

Agents Window は、複数の AI エージェントをローカル・クラウド・リモート SSH・Git worktree をまたいで並列管理するためのエージェント専用コントロールプレーンである。

Cursor 3 以前は「エディタにチャットを添えた」設計だった。Cursor 3 ではその関係が逆転し、エージェントが主役・エディタが補助という構成になった。Agents Window は、この転換を実現する新インタフェースである。Composer パネルは廃止され、すべてのエージェントセッションがこのウィンドウに集約されている。

Composer（コンポーザー）: Cursor 3 以前のチャット入力パネル。Cursor 3 で Agents Window に置き換えられた。
コントロールプレーン（Control Plane）: 分散システムの運用管理を担う層。AWS の ECS/EKS コントロールプレーンと同じ発想で、エージェントの起動・停止・監視・リダイレクトを一元管理する。

なぜ生まれたのか

Cursor 2 時代のエージェントは、エディタのサイドパネルで1タスクずつ動く設計だった。この設計には構造的な制約が3つあった。

1

1タスク1エージェントの逐次実行

長い実装タスクは終わるまで待つしかなかった。並列に別のタスクを走らせる手段がなく、エージェントを有効活用できていなかった。

2

ファイル競合によるマージコンフリクト

複数のチャットセッションが同じファイルを書き換えると、変更が衝突する。共有のワーキングツリー上で複数の変更を並行させることは本質的に危険だった。

3

PC に縛られた実行ライフサイクル

エディタを閉じるとセッションが終了する。数時間かかる大規模リファクタを「ながら作業」させる手段がなかった。

Agents Window はこれら3問題を並列実行・worktree 隔離・クラウド実行で解決する目的で設計された。リリースは 2026年4月2日（Cursor 3.0）。Enterprise 向けには2週間のロールアウト期間が設けられ、その後すべてのプランで利用可能になった。

Git worktree（ワークツリー）: 同一リポジトリを複数のディレクトリに同時チェックアウトする Git の機能（git worktree add）。ブランチごとに独立した作業ディレクトリを持てる。

まず理解する3つ

これだけ理解すれば Agents Window の本質が把握できる。

Local vs Cloud Agent

Local: ファイルシステム・LSP に直接アクセス。5〜15秒でdiff表示。デバッグ・即時確認向き。

Cloud（Background）: Cursor のインフラで実行。PC電源OFFでも継続。大規模・長時間タスク向き。GitHub 連携が必要。

Worktrees

各エージェントに独立した Git worktree を与える仕組み。ファイル競合ゼロで並列実行できる。

/worktree コマンドで起動。不要なら即破棄できる「使い捨てブランチ」として使う。

/multitask

1つの大きなゴールを複数の非同期サブエージェントに分解・並列実行するコマンド（Cursor 3.2 / 2026-04-24）。

「4モジュール × 4エージェント ＝ 2時間分の作業を30分で完了」という使い方が典型。

たとえ話 1: 「工場の製造ライン管制室」

工場には複数の製造ライン（エージェント）があり、それぞれ異なる製品（タスク）を並行して生産している。Agents Window は制御盤（SCADA）であり、各ラインのステータスを一覧表示し、問題が起きたラインだけ停止・リダイレクトできる。ライン1はリアルタイムで出荷（Local Agent）、ライン2は夜間無人自動生産（Cloud Agent）という使い分けがそのまま Local/Cloud の使い分けに対応する。

たとえ話 2: 「独立した実験室（worktree）」

研究室で同じ試料（コードベース）を複数の研究者（エージェント）が同時に実験する際、試料を使い回すと互いの変更が混ざる。worktree はそれぞれの研究者に専用の実験台と試料コピーを割り当てる仕組みだ。実験が失敗したら実験台ごと廃棄すればよく、元の試料は汚染されない。これが /worktree の本質だ。

たとえ話 3: 「バッチジョブと対話型プロセスの使い分け」

Linux の cron ジョブ（夜間バッチ）と対話型シェルの違いに対応する。Cloud Agent は nohup 付きのバッチジョブであり、セッション切断後も実行が続く。Local Agent は対話型シェルであり、端末に張り付いてリアルタイムで stdout を確認できる。両者を同時に持ち、素早い修正は Local で、長時間タスクは Cloud で、という使い分けが Agents Window 上で可能になる。

たとえ話 4: 「設計レビュー vs 実装フェーズ（Plan Mode）」

ウォーターフォールの設計フェーズにあたるのが Plan Mode（Shift+Tab で切り替え）だ。エージェントはコードを書く前に Markdown 形式の設計書を出力し、人間がレビューして承認する。実装フェーズに進んでから「方針が違った」と気付くと 40分の時間ロスになる——これを防ぐのが Plan Mode の役割である。

Remote SSH Agent について

アーキテクチャ図に示した「Remote SSH（リモート SSH）」は、自社サーバー・クラウド VM・HPC クラスタなどローカルでも Cursor クラウドでもない外部マシンにエージェントを接続して動かすモードです。

SSH（Secure Shell：セキュアシェル）: ネットワーク越しに別のコンピュータを安全に操作するプロトコル。Remote SSH エージェントは、社内サーバーや GPU クラスタ上のファイルに直接アクセスしながら、コードの生成・実行をローカルと同様に行える。クラウドエージェント（Cursor インフラ）が使えない環境や、特定のハードウェア・データが必要な場合に使う。

実践ベストプラクティス

1

ゴール指向プロンプトを書く

「42行目を修正して」ではなく「認証フローをトークン自動リフレッシュに変更し既存テストをパスさせよ」と書く。Cursor 3 以降、エージェントは自分でコンテキストを収集するため、手順より目的を渡した方が品質が上がる。

2

関連タスクをバッチ並列化する

「アプリ全体にテストを追加」という1エージェントへの指示ではなく、モジュール単位に4エージェントを立てて並列実行する。並列化が最大の時間乗数になる。

3

.cursor/rules を継続更新する

プロジェクトの規約・パターンをルールファイルに蓄積する。これが出力品質への最も効果的な介入だ。エージェントを「育てる」感覚で更新し続ける。

4

diff を全ファイルではなく差分として確認する

Agents Window の diffs view は変更箇所だけを提示する。レビューはファイル全体ではなく diff に集中する。変更されていない周辺コードは信頼してよい。

5

最大8並列を守り、監視コストを考慮する

同時起動可能な Background Agent は最大8。Tiled Layout は4〜5ペイン程度が実用限度。タイルが小さすぎると diff が読めなくなり監視が形骸化する。

初心者がはまりやすい落とし穴

エージェントモードに最初から飛びつく

Cursor 公式は「Tab 補完 → Inline Edit → Agent Mode」の順で習熟を推奨している。レビュー習慣が身についていない段階でエージェントに500行のコードを生成させると、何が変わったか把握できなくなる。

曖昧な指示でクラウドエージェントを起動する

Cloud Agent は PC を閉じても動き続けるため、方針が曖昧なまま起動すると数時間後に意図と異なる実装が出来上がる。複雑なタスクは Plan Mode で設計書をレビューしてから Cloud に投げること。

GitHub 連携なしで Cloud Agent を使おうとする

Cloud Agent（Background Agent）はリポジトリが GitHub 上にあることを前提とする。プライベートリポジトリや社内 GitLab 環境では直接使えない場合がある。

会話履歴の同期バグ（既知問題）

エディタの Agents パネルと Agents Window 間で会話履歴がリアルタイム同期しないことがある。「履歴が消えた」と感じたら Ctrl+Shift+P → Developer: Reload Window を試す。

.cursor/rules を一度作って放置する

ルールファイルはプロジェクト初期に一度書いて終わりではない。エージェントの出力をレビューするたびに、「ここはルール化できる」という学習を蓄積し続けることで品質が指数的に上がる。

専門用語集

Agents Window（エージェントウィンドウ）
Cursor 3 で導入されたエージェント専用の管理インタフェース。ローカル・クラウド・リモート SSH・worktree の全エージェントを一元管理する。Composer パネルの後継。
Background Agent（バックグラウンドエージェント）/ Cloud Agent
Cursor のクラウドインフラ上で実行されるエージェント。PC 電源 OFF でも継続し、最大8並列まで起動できる。GitHub 連携が前提。→ Local Agent と対比。
Git worktree（ワークツリー）
同一リポジトリを複数のディレクトリに同時チェックアウトする Git 機能。各エージェントに専用のワーキングディレクトリを与えることでファイル競合を防止する。
Composer 2（コンポーザー2）※ Composer パネルとは別物
Cursor 独自の AI モデルの名称。Cursor 3 の Agents Window でローカルエージェントが使うデフォルトモデル。コスト効率が高い（入力 $0.5/M トークン）。「Composer パネル（廃止された旧 UI）」とは全く異なる概念で、名前が似ているが意味は別。
Tiled Layout（タイルレイアウト）
Cursor 3.1 で追加された、複数エージェントのチャットを画面分割して並べて表示する機能。実用的なペイン数の目安は4〜5。
Design Mode（デザインモード）
Agents Window 内蔵のブラウザプレビュー上で UI 要素を直接選択・注釈できるモード。「このボタンを赤に」のような自然言語指示をドキュメント座標と紐付けてエージェントに渡す。
Plan Mode（プランモード）
Shift+Tab で切り替わるモード。エージェントはコードを書く前に Markdown 形式の設計書を出力し、人間のレビューと承認を待つ。複雑なタスクの方針ミスを防止する。
LSP（Language Server Protocol：言語サーバープロトコル）
IDE と言語解析ツールを接続するプロトコル。Local Agent は LSP に直接アクセスし、定義ジャンプ・型チェック・補完情報を活用できる。Cloud Agent は LSP にアクセスできない。
ACP（Agent Client Protocol：エージェントクライアントプロトコル）
Cursor エージェントと外部 IDE（JetBrains 等）を接続するプロトコル。2026年3月から JetBrains IDE 内で Cursor エージェントを利用できるようになった。
MCP（Model Context Protocol：モデルコンテキストプロトコル）
AI モデルに外部ツール（データベース・API・ファイルシステム等）を接続する標準プロトコル。Agents Window のエージェントは MCP 経由で外部サービスを操作できる。
.cursor/rules（カーソルルール）
プロジェクトルート直下に置く設定ファイル。コーディング規約・命名規則・禁止パターン等を記述し、エージェントがすべてのリクエストで参照する。品質の最大レバー。

次に学ぶと良いもの

Git worktree の詳細

git worktree add / list / remove の操作。worktree 上での rebase・merge 戦略。

MCP（Model Context Protocol）

エージェントに外部ツールを接続する標準仕様。Cursor のエージェントが DB・API・Slack と連携する仕組み。

Cursor Rules (.cursor/rules) 設計

ルールファイルの構造・スコープ設定・ファイルパターンマッチング。品質向上への最短経路。

GitHub Actions × Cloud Agent 連携

PR トリガーで Cloud Agent を自動起動するパターン。CI/CD の一部として AI エージェントを組み込む。

コンテキストエンジニアリング

エージェントへのコンテキスト供給設計。@ファイル/@フォルダ参照・ドキュメントインデックスの最適化。

Cursor SDK による自動化

TypeScript / Python SDK を使って CI・スクリプトから Cursor エージェントをプログラムで制御する。
