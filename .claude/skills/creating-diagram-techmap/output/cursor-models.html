ひとことで言えば

CursorはAnthropicのClaude・OpenAIのGPT・GoogleのGeminiなど複数社のLLMと、自社のComposer 2を束ねたIDE。タスクの性質に応じてモデルを選ぶことで、コスト・速度・品質の最適解を引き出せる。

なぜ生まれたのか

2023年〜：「1社独占」モデルの限界

CursorはVSCode（Visual Studio Code：Microsoftが開発するオープンソースのテキストエディタ）のフォーク（派生版）として開発されたAI特化IDEで、2023年に注目を集めた。当初はAnthropicやOpenAIなど特定のLLMに依存していた。しかし「コード補完に強い」「長いファイルの読み込みに強い」「推論が深い」など、用途によって最強モデルが異なるという現実が浮かび上がった。

VSCodeのフォークであるため、拡張機能・ショートカット・設定の多くはVSCodeと互換性がある。

解決策：マルチプロバイダー統合 + 自社モデル開発

Cursorはプロバイダーを束ねる「ルーター」として機能するよう設計を拡張し、さらにIDE内コーディングに特化したComposerという自社モデルを開発した。2026年3月にはComposer 2をリリース。同月のCursor 3では複数モデルを並列実行する/best-of-n機能も追加された。

背景にある構造的問題： LLMはすべてTransformerベースの確率モデルだが、学習データの構成・ファインチューニングの目的関数・RLHF（Reinforcement Learning from Human Feedback：人間のフィードバックによる強化学習）の報酬設計が異なるため、得意な出力分布が異なる。コーディングのような高度に構造化されたタスクでは、「どの強みを引き出すか」が生産性を大きく左右する。

まず理解する3つ

1

モデルは「プロバイダー × バリアント」で選ぶ

「Claude」という名前は会社（Anthropic）とモデルファミリーを指し、実際には「Claude Sonnet 4」「Claude Sonnet 4 Thinking」など複数のバリアントが存在する。Cursorの設定（Settings > Models）で現在利用可能なバリアントを確認するのが最も正確。

2

「Auto」は自動ルーティングであり「モデル名」ではない

Autoは特定のモデルではなく、Cursorがコンテキストに応じてコスト・速度・品質を考慮して最適なモデルを自動選択するシステム。Proプランでは月額料金枠の中でAutoが動作するため、プロンプトの性質次第で実際の消費コストが変わる。

3

Composer 2はコーディング専用・別課金プールで動く

Cursor自社モデルのComposer 2は、CursorBenchなどのコーディングベンチマークに特化して開発された。Proプランの通常の利用枠とは別の「スタンドアローン利用プール」として課金されるため、他のモデルを使い切ってもComposer 2は別途利用できる場合がある。

たとえ話で理解する

たとえ1：カメラレンズの使い分け

一眼カメラは「ボディ（Cursor IDE）」が同じでも、広角（Gemini：広い視野＝長コンテキスト）・望遠（GPT-5：精密なエージェント制御）・マクロ（Composer 2：IDE内コーディングへの極限特化）とレンズを交換することで全く異なる撮影能力を引き出せる。「最強レンズ」は撮る被写体によって変わる。

たとえ2：Autoはカーナビの「最速ルート自動選択」

「最速ルート」ボタンを押すと、カーナビが渋滞・距離・燃費を考慮して道を決める。AutoもCursorがコスト・速度・タスク性質を見てモデルを選ぶ。ルートは変わっても目的地には着く。ただし「料金所（課金）」を通る経路になることもある。

たとえ3：Extended Thinking は「ちょっと考えさせてください」の時間

ベテランエンジニアに難しい設計相談をすると、即答せず「少し設計書を書いてきます」と席を立つことがある。Extended Thinkingも同様に、最終回答を出す前に内部で仮説・反論・別アプローチを並走させ、その後に答えを絞り込む。この「内側の思考過程」がCursor UIでThinkingブロックとして折りたたみ表示される。

たとえ4：/best-of-n は「複数の候補図面を並べて比較」するプロセス

建築設計事務所で2〜3社の設計案を同時に依頼し、クライアントが最良案を選ぶようなプロセス。Cursor 3の/best-of-n機能は、複数モデルに同じプロンプトを並列実行させて最良の結果を選べる。品質を最大化したいが「どのモデルが正解か分からない」ときに有効。

ディープシンクモード（Extended Thinking）

Extended Thinking（拡張思考）: LLMが最終回答を出力する前に、内部で「仮説の立案→検証→反論→修正」のサイクルを反復する推論プロセス。Anthropic公式の用語はExtended Thinking。OpenAI系ではo系モデルの推論（Reasoning）が近い概念。

初心者がはまりやすい落とし穴

「Autoに任せれば最強モデルが使える」という誤解

AutoはコストバランスのためにPro枠内で動作する。プロンプトが複雑であれば高コストモデルを選ぶが、タスクによっては軽量モデルで済ませることもある。「コストを気にせず最強を使いたい」なら手動で明示指定する。

回避策: エージェントモードやアーキテクチャ相談は手動でComposer 2やGPT-5を指定する

Thinkingモードを常時オンにしてコストが爆発する

Extended Thinkingは内部思考分のトークンも課金対象になる。「簡単なバグ修正」や「リファクタリング」にThinkingを使うと不必要にコストがかかる。難易度と費用対効果を考慮して使い分ける必要がある。

回避策: 「正答率95%でいいなら通常モード、99%必要な設計判断ならThinking」のような基準を持つ

GPT-5に曖昧な指示を出してフラストレーションを溜める

Cursor公式ブログでもGPT-5は「指示が明確なほど真価を発揮する」と言及している。「なんかいい感じに」「うまくやって」といった曖昧な指示では意図と異なる出力になりやすい。

回避策: 「入力・出力・制約・やってはいけないこと」を4点セットでプロンプトに書く

ProプランとComposer 2の課金を混同する

Proプランの$20相当API枠とComposer 2のスタンドアローン利用プールは別管理。「Proの枠を使い切った」からといってComposer 2まで使えなくなるわけではなく、逆も然り。請求ダッシュボードで別々に確認する必要がある。

回避策: cursor.com/dashboard/usageで消費状況を定期的にチェックする

専門用語集

LLM
Large Language Model（大規模言語モデル）の略。大量のテキストデータで学習した確率モデルで、次のトークンを予測することで文章を生成する。Cursor内のすべての「モデル」がLLM。
Provider（プロバイダー）
LLMを提供する会社（Anthropic・OpenAI・Google・xAI・DeepSeekなど）。CursorはこれらのAPIを統合したマルチプロバイダーIDE。
Context Window（コンテキストウィンドウ）
モデルが1回の推論で参照できるトークン数の上限。大きいほど長いコードや複数ファイルを一度に読み込める。Geminiは業界最大級の長コンテキストが強み。
Extended Thinking（拡張思考）
Anthropicの公式用語。回答前に内部で多段階の推論を行うモード。Cursor UIではThinkingブロックとして折りたたみ表示。Claude 4系・Claude 3.7 Sonnetが対応。→ 関連: ディープシンクモード
TPS
Tokens Per Second（毎秒生成トークン数）。モデルの「速さ」を表す指標。Composer 2はこの値が高い設計。
Auto
Cursorがコンテキスト・コスト・速度を考慮して最適なモデルを自動選択するルーティング機能。特定のモデル名ではない点に注意。
SWE-bench
Software Engineering Benchmark（ソフトウェアエンジニアリングベンチマーク）の略。実際のGitHubイシューを解決できるか評価するコーディングAIの標準指標。Composer 2の性能評価に使用。
/best-of-n
Cursor 3で追加された機能。複数のモデルに同一プロンプトを並列実行し、最良の出力を選択できる。「どのモデルが正解か分からない」高品質要求タスクに有効。
MCP
Model Context Protocol（モデルコンテキストプロトコル）の略。外部ツール（ブラウザ・ファイルシステム・APIなど）をLLMに連携させるプロトコル。ProプランでCursorから利用可能。
Steerability（操舵性）
プロンプトの指示に対してモデルがどれほど正確に従えるかの特性。GPT-5の強みとしてCursor公式が言及。指示が明確であるほど真価を発揮する。
フロンティアモデル
現時点で最高水準の性能を持つLLM（Claude Sonnet 4・GPT-5・Gemini Ultra等）の総称。Proプランの$20枠で利用できる対象モデル群。軽量モデル（mini系）と対比される。
Tab補完
カーソル位置を基にAIがコードの続きを予測してサジェストし、Tabキーで確定する機能。コード補完の代表的な操作。Cursorでは高速な軽量モデルが担当する。
Hooks（フック）
特定のイベント（ファイル保存・コマンド実行など）をトリガーに処理を自動実行する仕組み。CursorのProプランで利用できるワークフロー自動化機能。
SSO
Single Sign-On（シングルサインオン）の略。1回の認証で複数システムにアクセスできる仕組み。CursorのTeams/Enterpriseプランで提供される企業向け認証機能。
CursorBench
Cursor社が開発したコーディングAIの評価ベンチマーク。実際のIDEでのコード編集・デバッグ・リファクタリングタスクを数値化する。Composer 2の優位性評価に使用。→ 関連: SWE-bench
