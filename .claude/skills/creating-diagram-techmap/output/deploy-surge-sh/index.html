Techmap

surge.sh

静的ファイルをコマンド一発でCDN公開する、フロントエンド向けホスティングサービス

ひとことで言えば

「npx surge dist/ myapp.surge.sh」の一行で、ビルド済みHTMLをHTTPS付きのCDNに公開できる静的ホスティングサービス。

サーバー管理・DNS設定・SSL証明書の取得——これらを一切意識せず、ローカルのフォルダをそのまま全世界に公開する体験を提供する。対象はHTML/CSS/JS/画像などの静的ファイルのみで、PHPやNode.jsのサーバーサイド処理は動かない。

対象読者フロントエンドエンジニア・デモ公開を急ぐ開発者
得意なこと静的ファイルの即時公開・プロトタイプ共有
できないことサーバーサイド処理・DB・サーバーレス関数

なぜ生まれたのか

2015年、Chloi Inc.（Harpなど静的サイトツールを手がけるチーム）が公開した。当時のフロントエンド開発では、Grunt・Gulp・Browserify・npm scripts（2015年当時主流だったJavaScriptビルドツール群）によるビルドフローが成熟しつつあったが、「そのビルド成果物を本番品質のCDNに出す手段」が整っていなかった。

既存の代替手段の不満点（2015年当時）

  • GitHub Pages：Gitリポジトリとの結合が前提。任意フォルダを今すぐ出す使い方には弱い
  • Amazon S3：バケット設定・IAM・CloudFront設定が必要で手数が多い
  • クライアントサイドルーティング（SPA）の 404フォールバック や PushState 対応が手動設定必須

Surgeが解決したのは、クリーンURL・404カスタム・SPA対応・Gzip・CDN配信・HTTPSを、コマンド一発のワークフローに凝縮することだった。思想的背景として、創業者はCordova（PhoneGap）の共同開発者でもあり、オープンWebへのコミットメントが公式に明記されている。

npm run build
→
surge dist/
→
Surge CDN
→
ブラウザ

ビルド → デプロイ → 配信の最小ワークフロー

まず理解する3つ

この3点を押さえれば、surge.shの実用は9割カバーできる。

1 コマンド一発でデプロイできる

npx surge [フォルダ] [ドメイン] で完結する。gitリポジトリが不要で、アカウント登録後は任意のフォルダを指定して即公開できる。

2 静的ファイルしかホストできない

HTML・CSS・JS・画像のみ対象。Node.js/PHP/Pythonのサーバープロセスは起動できない。「クライアントで完結するアプリケーション」が使用前提となる。

3 SPAは 200.html が必要

React Router等で /user/123 のような深いURLを直打ちすると、デフォルトでは404になる。200.html（index.htmlのコピー）を置くと全URLをキャッチオールでき、SPAが正常動作する。

主要概念・アーキテクチャ

基本ワークフローとCLIコマンド

以下のコードが行うこと：surge CLIのインストールから、デプロイ・削除までの主要コマンド

# インストール（グローバル）
npm install -g surge

# 初回：アカウント作成を促すプロンプトが出る
surge

# 特定フォルダ・ドメインを指定してデプロイ
surge dist/ myapp.surge.sh

# npx でインストール不要（CI/スクリプトに向く）
npx --yes surge dist/ myapp.surge.sh

# 公開サイト一覧
surge list

# サイトを削除
surge teardown myapp.surge.sh
CLI（Command Line Interface：コマンドライン インターフェース）
ターミナル（黒い画面）からテキストコマンドでツールを操作する仕組み。GUIとは対称的で、自動化・CI組み込みに適している。

Surge CDNの内部構造：どうファイルが届くのか

Surgeは商用CDNに依存せず、自前のSurge CDNを構築・運用している（公式: "About the Surge CDN"）。ファイルのアップロード時に各ファイルへETag（Entity Tag：エンティティ タグ）と呼ばれる識別子が付与され、世界10リージョンのエッジノードへ配信される。

ETagを使った差分アップロード→エッジ配信の流れ

CDN（Content Delivery Network：コンテンツ デリバリー ネットワーク）
地理的に分散した複数のサーバー（エッジノード）からコンテンツを配信するネットワーク。ユーザーの最寄りノードから応答するため、レイテンシが低減する。
ETag（Entity Tag：エンティティ タグ）
HTTPヘッダの一種。リソースの特定バージョンを識別するハッシュ値的な文字列。ブラウザが保持しており、次回リクエスト時に「手元のバージョンが最新か」をサーバーに確認できる。ファイルが変わっていなければ本体を再送しない（304 Not Modified）。

Lucid Caching：自動キャッシュ戦略

SurgeはLucid Cachingと呼ぶキャッシュ制御を自動で適用する。ファイルの内容変化をETagで検出し、変わっていないファイルは再配信しない。変わった場合は新しいETagで配信する。加えて、静的アセット（JS・CSS等）にはFar-future Expiresヘッダ（長期キャッシュ）を付与し、GZip圧縮で転送量を削減する。

たとえ話で理解する

たとえ1：手紙をポストに投函する
surgeコマンドを叩くのは、書いた手紙（HTMLファイル群）をポストに投函するようなもの。投函後は郵便局（Surge CDN）が各地の配達拠点（エッジノード）へ仕分けして届ける。宛先（ドメイン）は自分で書く。手紙の内容を変えたければ新しい手紙を投函するだけで、古い手紙は自動で差し替わる。
たとえ2：200.html＝ビルの交換台オペレーター
SPAで200.htmlを置かない場合、「101号室に来てください」という案内（URL）でビルを訪れた来客が「そんな部屋はない」と玄関で追い返される状態になる。200.htmlは「どの部屋番号で来ても、まずロビー（index.html）に通す交換台オペレーター」として機能する。部屋割り（ルーティング）は入館後にJavaScriptが判断する。
たとえ3：ETag＝商品の「品番シール」照合
ブラウザはファイルを受け取る際に、サーバーが発行した「品番シール」（ETag文字列）を手帳（キャッシュ）に控えておく。次回アクセス時、「この品番のものが手元にあるが、まだ最新か？」とサーバーに問い合わせる（If-None-Matchヘッダ）。品番が変わっていなければ「それが最新です（304）」と返すだけで本体を再送しない。ファイルの内容が変わった時だけ新しい品番で本体を再送する。これがLucid Cachingの核心動作。
たとえ4：.surgeignore＝「持出禁止リスト」
機密書類を会議室の外に持ち出さないためのルールシートと同じ構造。.surgeignoreに書いたファイルはアップロード対象から除外される。.git/・node_modules/・.envなどを明示しないとAPIキーや開発ログが丸ごと公開されるリスクがある。

挙動を制御する特殊ファイル

ファイル名 役割 使いどころ
200.html 全URLを200として返すキャッチオール SPAでReact Router等を使う場合
404.html カスタム404ページ 存在しないURLへのアクセス時

カスタムドメインの設定方法

カスタムドメインは無料プランでも使用できる。DNS設定とSurge側の指定の2ステップで完了する。

ステップ 作業 具体的な設定値（例）
①DNS設定 ドメインレジストラのDNS管理でCNAMEを追加 www → na-west1.surge.sh
②Surge側 デプロイ時にカスタムドメインを指定 surge dist/ www.example.com

以下のコードが行うこと：CNAMEファイルにドメインを書いておくことで、次回以降のデプロイでドメイン指定を省略できる

# dist/CNAMEファイルにドメインを書く
echo www.example.com > dist/CNAME

# 以降はドメイン指定なしでデプロイ可（CNAMEから自動読み込み）
surge dist/
CNAME（Canonical Name：正規名）レコードの補足
surgeの場合、CNAMEレコードの向け先（surgeのホスト名）はna-west1.surge.shが一般的に使われているが、最新の推奨値は 公式ドキュメント で確認すること。Apex（ルート）ドメイン（例: example.com そのもの、wwwなし）の場合はCNAMEではなくAレコードが必要なケースもある。

CI/CDへの組み込み

以下のコードが行うこと：GitHub ActionsからSurgeへ自動デプロイする設定例

# .github/workflows/deploy.yml（抜粋）
- name: Deploy to Surge
  env:
    SURGE_LOGIN: ${{ secrets.SURGE_LOGIN }}
    SURGE_TOKEN: ${{ secrets.SURGE_TOKEN }}
  run: npx --yes surge dist/ myapp.surge.sh

SURGE_LOGIN（メールアドレス）とSURGE_TOKENをリポジトリのSecretsに登録することで、対話なしに自動デプロイできる。 --yesフラグはCLIの対話プロンプト（「このフォルダでよいか？」等の確認）を自動的にスキップするためのオプションで、CI環境では必須。

SURGE_TOKENの取得方法：

# ログイン済みの状態でトークンを表示
surge token
# → 表示された文字列をGitHub Secrets の SURGE_TOKEN に登録
SPA（Single Page Application：シングル ページ アプリケーション）
単一のHTMLファイルを読み込んだ後、JavaScriptがDOMを動的に書き換えてページ遷移をエミュレートするアーキテクチャ。React・Vue・Angularで一般的。URLの変化はPushState APIで実現し、サーバーへの追加リクエストを発生させない。
PushState API
ブラウザのHistory APIの一部。JavaScriptからURLバーの表示を変更できる（ページリロードなし）。history.pushState()で実現。SPAのルーティングの根幹を支える。

初心者がはまりやすい落とし穴

SPA の深いURLが 404 になる

状況：React Routerで /user/123 にリンクして共有したら、相手が直打ちすると404になる。
原因：Surgeはデフォルトでリクエストパスに対応するファイルを探す。/user/123.htmlが存在しないため404を返す。
対策：index.htmlを200.htmlにコピーして一緒にアップロードする。cp dist/index.html dist/200.html をビルドスクリプトに追加。

更新が反映されない（キャッシュ問題）

状況：surgeを実行して「アップロード完了」と表示されたのに、ブラウザを開くと古い表示のまま。
原因：Lucid Cachingで強いキャッシュが付いている。ETagが変わっていないとブラウザは手元のキャッシュを使い続ける。
対策：ブラウザのハードリロード（Ctrl+Shift+R）か、シークレットウィンドウで確認。また、JSやCSSのファイル名にコンテンツハッシュ（app.abc123.jsのような命名）を含めるビルド設定が根本的な解決策。

秘密情報が公開される

状況：surge .（カレントディレクトリ全体）を実行したら、.envファイルが公開されてAPIキーが漏洩した。
原因：デフォルトで.git/・node_modules/は除外されるが、.envは除外されない。
対策：.surgeignoreに.env・*.pem・秘密情報を含むファイルを明示的に追加する。ビルド成果物のみのフォルダ（dist/やbuild/）を対象にするのがベストプラクティス。

カスタムドメインの独自SSLは無料ではない

誤解：「カスタムドメインが無料ならHTTPSも当然ついてくる」と思ったが、独自ドメインでの本格的なSSL証明書・HTTP→HTTPS強制は有料プラン（Surge Professional）が必要。
実態：*.surge.shサブドメインはBasic SSLで無料HTTPS対応。独自ドメインの厳格なTLS運用には月額 $30 のプロプランが必要。プロトタイプ・内部共有ならsurge.shサブドメインで十分なケースが多い。

surge teardownが期待通りに動かないことがある

状況：surge teardown myapp.surge.shを実行しても、しばらくはURLにアクセスできる・エラーが出る、といった報告が GitHub Issuesに複数存在する。
原因：CDNキャッシュの伝播（プロパゲーション）には時間がかかる。コマンドの完了≠全エッジノードへの削除反映ではない。
対策：削除は「一度で確実」と過信しないこと。数分待ってからシークレットウィンドウで確認する。

専門用語集

CDN（Content Delivery Network：コンテンツ デリバリー ネットワーク）

地理的に分散した複数のエッジサーバーからコンテンツを配信する仕組み。ユーザーに近いサーバーが応答するためレイテンシが低い。→関連: エッジノード、Lucid Caching

CLI（Command Line Interface：コマンドライン インターフェース）

ターミナルからテキストコマンドで操作するインターフェース。Surge CLIはnpm install -g surgeで入れる。→関連: npx、CI/CD

SPA（Single Page Application：シングル ページ アプリケーション）

単一HTMLを読み込んでJavaScriptがDOMを動的に書き換えるアーキテクチャ。URLはPushState APIで変化させる。Surgeでは200.htmlが必要。→関連: PushState、200.html

ETag（Entity Tag：エンティティ タグ）

HTTPレスポンスヘッダ。リソースのバージョン識別子。ブラウザは次回リクエスト時にETagをサーバーに送り、変化がなければ304（Not Modified）で本体を再送しない。→関連: Lucid Caching、キャッシュ

Lucid Caching

SurgeのCDNが採用するキャッシュ制御戦略。ETagによる差分検出・Far-future Expiresヘッダ・Gzip圧縮を組み合わせる。→関連: ETag、CDN

200.html

Surge固有の特殊ファイル。このファイルが存在すると、マッチするファイルが見つからないURLすべてに対してHTTP 200でこのファイルを返す（キャッチオール）。SPAのクライアントサイドルーティングで必須。→関連: SPA、PushState

CNAME（Canonical Name：正規名）

①DNSレコードの一種。ドメインを別のドメイン（エイリアス）に解決する。カスタムドメイン設定で使用。②Surgeプロジェクト内のCNAMEファイル：デプロイ先ドメインを記憶させ、再デプロイ時に省略可能にする。

静的ホスティング（Static Hosting）

サーバーがHTMLやファイルをそのまま配信するモデル。PHPやNode.jsのようなサーバーサイド処理は行わない。ビルド済みの成果物を配信するJamstackアーキテクチャの基盤。→関連: Jamstack

npx

npmに同梱されたコマンド実行ツール。パッケージをグローバルインストールせずに一時実行できる（npx surgeなど）。CI環境での利用に向く。→関連: CLI、CI/CD

クリーンURL（Clean URL）

URLに拡張子を含めない表記スタイル。/about.htmlではなく/aboutのようにアクセスできる状態。SurgeはデフォルトでクリーンURLに対応しており、about.htmlが存在すれば/aboutでもアクセスできる。→関連: 静的ホスティング

Far-future Expires（長期Expiresヘッダ）

HTTPレスポンスヘッダの一種。ブラウザに「このリソースは非常に長期間（例：1年後まで）有効」と伝え、期間内は再リクエストなしにキャッシュを使わせる。`Cache-Control: max-age=31536000`として表現されることが多い。Surgeは静的アセットに自動で適用する。→関連: Lucid Caching、ETag

Jamstack

JavaScript・API・Markupの頭文字。事前ビルド済み静的ファイル＋外部APIの組み合わせでWebアプリを構成するアーキテクチャ思想。Surgeはその配信層（ホスティング）を担う。→関連: 静的ホスティング、CDN

CORS（Cross-Origin Resource Sharing：クロス オリジン リソース シェアリング）

異なるオリジン（ドメイン）間のリソース共有を制御するHTTP仕組み。CORSヘッダのカスタム設定はSurge Professionalで可能。→関連: HTTP、セキュリティ
