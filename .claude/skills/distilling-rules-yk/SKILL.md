# Rule Distiller（コード→ルール蒸留）

セッションで踏んだ実装上の気づきを、次回以降に役立つ形でルールへ落とし込む。

## 追記する価値がある気づき（どれか 1 つ満たせば追記）

| 種別 | 判断基準 |
|------|---------|
| バグ根本原因 | デバッグに時間がかかった · 次回また踏みそう |
| 設計パターン | 知っていれば 30 分節約できた · 非自明な選択 |
| 実運用の注意点 | 公式に書いていない · SSR / 環境依存の落とし穴 |

## 追記しないもの

- コード・公式ドキュメントを読めば自明なこと
- プロジェクト固有の一時決定（handoff に書く）
- 既にルールに書いてあること

## 手順

### Step 1: 気づきを列挙する

今回のセッションで判明したことを「次回同じバグを踏まないために」「知っていれば時間を節約できたこと」で絞って箇条書きにする。ユーザーに確認してから進めてもよい。

### Step 2: ドメインを特定し L1 ファイルを選ぶ

`C:\yk-skill\rule\RULE_INDEX.md` のタスク別クイック入口から L1 を特定する。複数ドメインにまたがる場合はそれぞれに追記する。

| 気づきのドメイン | L1 ファイル |
|-----------------|------------|
| React Hooks · コンポーネントパターン | `REACT_RULES.md` |
| Next.js / SSR · API Routes | `NEXTJS_RULES.md` |
| Tailwind レイアウト | `TAILWINDCSS_RULES.md` |
| Git 運用 | `GIT_WORKFLOW_RULES.md` |
| スキル設計 | `SKILL_AUTHORING_RULES.md` |

### Step 3: L1 ファイルを Read して追記位置を決める

- 既存セクション末尾が自然な場合 → そこに追記
- 新セクションが必要な場合 → `### YK パターン補足` 等を追加

### Step 4: 追記する

「なぜ必要か」「何がアンチパターンか」「どう直すか」を 30 秒で読める量に絞る。コードブロックは最小限（説明が不要なら省く）。

### Step 5: ユーザーに報告する

追記したファイル · 内容の要点（1〜2 行/件）を伝える。
