# M1 整合 — 手順と判定表

## 手順

### Step 1: Tier とプロジェクト

[scope-tiers.md](scope-tiers.md) で Tier を決める。slug / リポを特定（不明なら 1 問）。

### Step 2: 機械チェック（Grep）

Tier P 範囲で次を検索する。

| パターン | 目的 |
|----------|------|
| `未着手` · `実装未` · `次:` · `次 §` | ステータス古さ |
| `合意済` · `完了` · `実装済` | 対照用 |
| セッションで触った機能名（例: 表ペイン · 列幅） | 横断漏れ |

### Step 3: コード・Git 照合（機械的矛盾）

実装完了の主張がある資料について:

- 該当パスを Grep（コンポーネント名 · ボタンラベル · export 名）
- 直近 `git log` は **ユーザーが Shell 明示時のみ**（通常はセッション文脈 · 既知コミット hash でよい）

### Step 4: 判定と対応

下表の **種別** に従う。

### Step 5: 報告テンプレ

```markdown
[整合] {project}（Tier {P|P+H|C}）

## 1 行サマリー
{修正 N 件 · WARN M 件}

## 即修正したもの（機械的）
| ID | ファイル | 変更要点 |
|----|----------|----------|

## 報告のみ（意味的 — 要判断）
| ID | ファイル | 内容 | 提案 |
|----|----------|------|------|

## 触っていないが確認済み
- {ファイル一覧}

## 次にできること
- 「コミットして」（committing-with-git-yk）
- Tier C 横断チェック（明示依頼時）
- M2 資料最新化（別モード · refresh.md）
```

## チェック ID と判定

| ID | 検出内容 | 種別 | 対応 |
|----|----------|------|------|
| S1 | 方針 MD が `実装: 未着手` だがコード・ボタン一覧・コミットで実装済み | **機械的** | 方針 · UI仕様 · ロードマップのステータスを `実装済` / 日付に更新 |
| S2 | `decision-log` が `次: §4 実装` だが該当機能が完了 | **機械的** | decision-log 行を `完了` + コミット hash または日付 |
| S3 | `現状とロードマップ` の「次」が完了済みタスクのまま | **機械的** | 次タスクを HANDOFF §6 または未完了項に合わせる |
| S4 | `ボタン一覧` の実装状態と UI コード不一致 | **機械的** | ボタン一覧の表をコードに合わせる |
| S5 | `UI仕様` が方針 MD とステータス不一致 | **機械的** | 方針 SSOT に合わせる（`PROJECT_DOCUMENT_RULES` §9） |
| S6 | 同一画面の記述が 2 ファイルで矛盾（どちらが正本か明確） | **機械的** | scope-tiers の優先順で追随側を修正 |
| M1 | 方針 A と方針 B が同ランクで食い違い | **意味的** | 報告 · grill-me またはユーザー判断を促す |
| M2 | 「未着手」が次フェーズ用として意図的か不明 | **意味的** | 報告 · 意図確認 |
| M3 | 削除 vs アーカイブの判断が要る | **意味的** | 報告（M2 更新モードへ誘導可） |
| H1 | HANDOFF 最新セッションと README 不一致 | **機械的**（Tier P+H 以上） | `handoff-session-work` folder-audit R1 に準拠 |

**種別の目安**

- **機械的**: 単一の正本がコードまたは SSOT 方針 MD で確定できる
- **意味的**: プロダクト判断 · 履歴の残し方 · 2 つの Accepted 方針の競合

---

## 修正時のルール

- **最小差分** — ステータス行 · 表の1行 · 日付 footnote 程度を優先
- **ADR 本文の書き換え禁止** — 経緯は decision-log 1 行で追う
- **handoffs 本文** — Tier P では触らない。Tier P+H で H1 のみ（終了ゲートは handoff-session-work）
