# レビュー視点カタログ

マルチエージェントでは、**異なる視点からのフィードバック**が単一レビューより品質改善に寄与しやすい（多角的探索・専門化した批評役の割当が有効、という知見が文献・実務で繰り返される）。

## 視点の選び方（先に決める）

1. **レビュー対象の種類**：ドキュメント / コード / UI・体験 / スキル・ルール / 設計・API など。
2. **失敗時のコスト**：セキュリティ・法令・金銭・評判・運用停止のどれが効くか。
3. **読者・利用者**：誰が最終的にその成果物を使うか。

上記から **最低 3 つ**、重複しないレンズを選ぶ（同じ立場を 2 体にしない）。

## レンズ一覧（状況に応じて組み合わせ）

| レンズ | 何を見るか | 採りやすい状況 |
|--------|------------|----------------|
| **報告者（提示側）** | 主張の筋・根拠・省略・読者への誘導は妥当か | 説明・報告・提案文・PR 説明 |
| **報告先（受け手）** | この文面・UI・API を受け取った人が誤解・作業不能にならないか | オンボーディング・Runbook・仕様書 |
| **第三者（中立監査）** | 前提の抜け・論点のすり替え・過剰断定・バイアス | どの成果物にも有効な「外野」 |
| **プロダクトエンジニア / Staff** | 正しさ・保守性・パフォーマンス・境界条件・テスト可能性 | コード・設計・リファクタ |
| **セキュリティ / 悪用者視点** | 入力・権限・秘密情報・依存のリスク | 認証・公開 API・スクリプト・MCP |
| **SRE / 運用** | デプロイ・監視・ロールバック・障害時の手順の足りなさ | インフラ・CI・Runbook |
| **IxD / UX（体験設計）** | タスク達成・認知負荷・エラー回復・一貫性 | 画面・CLI・ドキュメントの利用フロー |
| **ビジュアル / UI デザイン** | タイポグラフィ・余白・コントラスト・コンポーネント規約 | フロント・デザインシステム連携 |
| **PM / ステークホルダー** | 目的・スコープ・優先順位・「やらないこと」の明確さ | 仕様・ロードマップ・要求整理 |
| **コンプライアンス / 品質監査** | 禁止表現・ログ・個人情報・証跡 | 外向き文書・ログ設計 |
| **レッドチーム / 悪魔の代弁** | 反論・極端ケース・「意図的に壊す」読み | 重要判断・セキュリティ・説得力が要る提案 |

## トリオ例（そのまま使える組み合わせ）

- **コミュニケーション**：報告者 + 報告先 + 第三者
- **スキル・ルール文書**：執筆者意図（報告者に近い）+ 適用するエージェント側（報告先）+ SSoT・簡潔さ（第三者／監査）
- **UI 変更**：IxD/UX + フロント実装（Staff）+ アクセシビリティまたはビジュアル（どちらか一方＋第三者）
- **セキュリティ敏感**：Staff + セキュリティ + レッドチーム

## サブエージェントへの依頼で必ず含める要素

各サブエージェントは**ユーザーの生メッセージにアクセスできない**。親が **対象テキストまたはパス・制約・用語** をプロンプトに埋め込む。

- レビュー対象の**全文またはファイルパス**
- **採用レンズ 1 つ**（その視点以外は原則出さない／最後に「他レンズに任せる」と明記）
- **出力形式（順序固定）**：
  1. **`## 採用レンズ`** … レンズ名（カタログの表記と一致させる）
  2. **`## この視点の内容`** … そのレンズが**何を見るか・何を見ないか**を 1〜3 文で書く（上の「レンズ一覧」表の「何を見るか」を要約してよい。カスタムレンズなら親がプロンプトで渡した定義を言い換えて明示する）
  3. **`## 所見`** … 箇条書き、重大度、根拠、**修正提案は具体的に**
- **禁止**：他レンズの全面代行、実装の勝手な変更（読み取りレビューなら readonly 明示）、**視点の開示なしで所見だけ出すこと**
