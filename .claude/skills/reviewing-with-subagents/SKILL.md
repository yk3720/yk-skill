# サブエージェントによる多視点レビュー

Cursor の **`Task` ツールでサブエージェントを並列起動**してレビューさせる手順（ユーザーが「エージェントを立ち上げてレビュー」と言う場合の想定運用）。

**単一パスのコードレビュー**（差分・PR 前・1 エージェント 1 周）→ `reviewing-code-yk`。

## 目的

同じ成果物に対し **3 体以上のサブエージェント**を**同一ターンで並列起動**し、**互いに異なるレビュー視点**だけを担当させる。親エージェントが結果を統合する。

### なぜ視点を分けると効率がいいか（調査に基づく要約）

- **単一焦点**：各レビュアーに「何を見るか」を 1 系統に固定すると、指摘が浅くなりにくく、**観点の取りこぼしが別レンズで補完**されやすい（委員会型・多エージェントのコードレビュー設計で繰り返されるパターン）。
- **欠陥の重複が減る**：正しさ・セキュリティ・体裁など **異なるレンズは表面化しやすい問題の分布が重なりにくい**ため、並列の専門レビュアー構成は網羅性に有利、という整理が多い。
- **文書・報告書では役割分離が特に有効**：**技術的正確性**（事実・手順・数値が合うか）と**編集・様式**（読みやすさ・テンプレ準拠）と**コンプライアンス**は、同じ人に全部任せるより **別レンズに分けた方がボトルネックと見落としを減らせる**、という実務上のガイダンスがある。
- **コストとのトレードオフ**：変更が極小・機械チェックで足りる場合は並列レビューを避け、**人間／エージェントのレビューは高リスク・高影響・曖昧さが残る箇所**に寄せる。

## 前提（必ず守る）

- **並列数**：レビュー用途では **3 体以上**（ユーザーが少数を明示した場合は従う）。
- **1 体 1 レンズ**：各サブエージェントに **1 つの視点**を固定し、万能レビュアーにしない。プロンプトには **合格基準に近い「このレンズでは何を見て、何は見ないか」**を短く明記する。
- **文脈の埋め込み**：サブエージェントはユーザーメッセージを直接見られない。**対象全文またはパス・制約・用語**を各 `Task` の `prompt` に書き切る。
- **読み取りのみ**：ファイル変更が不要なら `subagent_type`: `explore` と `readonly`: `true` を検討する。
- **視点の開示**：各サブエージェントの報告の冒頭に、**どのレンズでレビューしたか**と、**そのレンズが何を見る視点か**（1〜3 文でよい）を必ず書かせる。親がユーザーへ返す最終報告でも、**レンズ名＋視点の要約**を所見と対応づけて示す。

## 効率化のための事前整理（起動前に親がやること）

- **機械化できる指摘はサブに渡さない**：フォーマッタ・リンター・スペル・CI で拾える体裁は先に潰し、サブは **判断・文脈・リスク**に集中させる（観点の無駄重複を減らす）。
- **成果物タイプでレンズを束ねる**：コード・報告書・スキル MD など種類ごとに「よく効くトリオ」を **[perspectives.md](references/perspectives.md)** から選び、**同じ立場のレンズを二重に置かない**。
- **報告書・外向き文書**では次の 3 系統の分離を特に推奨（詳細は perspectives の表）：**技術的正確性（SME）**／**編集・様式・テンプレ準拠**／**コンプライアンスまたは第三者監査**。
- **コード・設計**では例：**正しさ・境界条件**（Staff に近い）／**セキュリティ**／**運用・失敗時**（SRE）や **UX・利用者** など、**欠陥の種類が被りにくい組み合わせ**を選ぶ。

## 手順

### Step 1: 対象と制約の確定

親が把握する：レビュー対象（パス／貼り付け）、目的（マージ可否・品質・安全・UX 等）、禁止事項（例：スコープ外のリファクタ禁止）。

スキル文書をレビューする場合は `c:/yk-skill/rule/10_meta/SKILL_AUTHORING_RULES.md` を基準にできる（パスをプロンプトに含める）。

### Step 2: 視点の選定

→ [references/perspectives.md](references/perspectives.md) を読み、対象の種類とリスクに合わせて **3 つ以上のレンズ**を選ぶ。  
例：報告者・報告先・第三者／Staff・IxD・セキュリティ など。**同じ立場を二重に置かない。**

### Step 3: 並列起動

**1 回の応答で** `Task` を複数回並列呼び出す。各呼び出しで：

- `description`：短い識別名（例：`UX review`）
- `subagent_type`：`generalPurpose`（変更・調査が要る）または `explore`（読み取り中心）
- `prompt`：**レンズ名** + **その視点の定義（何を見るか）をプロンプト内に明記** + 対象 + **出力形式（下記）** + 他レンズへの越権禁止

各サブへの `prompt` に、次の**報告フォーマットをそのまま指示**する：

```markdown
## 採用レンズ
（例：報告先（受け手））

## この視点の内容
（このレンズが何を重視し、何を見ないか。1〜3 文。カタログの説明を要約してよい）

## 所見
（重大度・根拠・具体修正案）
```

### Step 4: 統合（親）

- 対立する所見は **根拠の強さ・重大度・スコープ**で整理し、単純平均しない。
- 各サブの出力は **重大度・根拠・修正案がセット**になっているかを確認し、曖昧な指摘は親が **優先度付け**する（オーケストレーター役）。
- ユーザーには **合意点 / 対立点 / 推奨次アクション** を簡潔に返す。
- 統合文でも **サブごとに「レンズ名 → 視点の要約 → 主な所見」**が追えるようにする（視点が混ざって読めない状態にしない）。

## アンチパターン

- サブエージェントを **1 体だけ**で「多角的」と称する。
- プロンプトが **「なんでもレビューして」** で、視点が重複する。
- サブ結果を **貼るだけ**で親が判断・優先順位付けをしない。
- **採用レンズ**や**視点の内容**を省略した報告（どの立場の批評か判別不能なまま所見だけ並べる）。
