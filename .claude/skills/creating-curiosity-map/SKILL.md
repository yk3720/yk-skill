# CuriosityMap

文系の大人が「今日初めてこのテーマを調べた」状態から、本質を自分の言葉で語れるようになるHTMLを生成し、surge.shに公開する。

## 対象読者

- **文系・非理系の大人**（専門的な理数系の素養は前提としない）
- テーマそのものには未経験だが、**日常の経験や常識と結びつけることで理解できる**タイプ
- 「なんとなく気になっている」から「人に話せるレベル」まで引き上げたい

> **前提として持っている知識**: 日常生活の経験・一般常識・中学校レベルの基礎知識のみ。数学・理科・プログラミングの専門知識は前提にしない。

## ゴール

> テーマの本質を「自分の日常」と結びつけて理解し、誰かに話せる・次に何を調べれば良いかわかる状態になること

---

## ワークフロー

### Phase 0: リサーチ（サブエージェント）

**exploreサブエージェント**を起動して対象トピックを徹底調査する。

```
Task({
  subagent_type: "explore",
  description: "テーマの徹底調査",
  prompt: `
    以下のテーマについて徹底的に調査してください：
    【対象】{トピック名}

    調査項目：
    1. このテーマがなぜ注目されているのか（背景・歴史・きっかけ）
    2. このテーマは私たちの生活とどう関係しているか（日常との接点）
    3. 主要な概念・用語の一覧（初めて見る人がまず知るべきもの）
    4. 身近な具体例・実例（生活の中で見かけるシーン）
    5. よくある誤解・思い込み（実際には違うこと）
    6. 料金プラン・ライセンス体系（ツール・サービスの場合のみ。無料枠・有料プランの違いを含む）
    7. 関連する公式ドキュメント・参考サイトのURL（できるだけ複数）
    8. このテーマに関連する周辺トピック・発展的な内容

    結果は箇条書きで整理して返してください。
  `
})
```

### Phase 1: ソース読み込み

1. URLなら `WebFetch` で取得、ドキュメントなら内容を分析
2. **情報の全体像を把握**（漏れなく）
3. Phase 0のリサーチ結果と照合して抜け漏れを確認
4. 「文系の大人が初めてこのテーマを見た場合」の視点で整理する

### Phase 2: 用語対応表を作成

専門用語を、読者が日常で知っている体験・感覚に結びつけて説明する。  
「難しい言葉をやさしくする」のではなく、**日常の経験に接続する**。

**略称・英語表記のルール**: 英語の略称や専門用語は、**初出時に正式名称と日本語での意味を必ず添える**。  
（例：AI → Artificial Intelligence（人工知能）、GDP → Gross Domestic Product（国内総生産））

| 用語 | 正式名称（略称の場合） | 一言の定義 | 日常の経験との対応 |
|------|---------------------|-----------|--------------------|
| （例）インフレ | Inflation（インフレーション） | モノの値段が全体的に上がること | 去年より今年のランチ代が高くなっている、あの感覚 |

### Phase 3: たとえ話を3つ以上考える

**たとえ話は「日常生活の中にあるもの」だけを使う。**  
技術・数学・プログラミングを使ったたとえは使わない。  
ただし正確性は損なわないこと。たとえが示す仕組みと実際の仕組みが対応しているか確認する。

**良いたとえ話の条件**:
- 「なるほど、そういうことか」が1〜2文で伝わる
- 読んだ人が「これ知り合いに話せるな」と思えるくらい具体的
- テーマへの誤解を生まない（正確性を犠牲にしない）

| 種類 | 例 |
|-----|----|
| 買い物・お金 | スーパーの値引きシール、クレジットカードの仕組み、ポイント還元 |
| 料理・食事 | レシピと材料の関係、調味料のバランス、火加減の調整 |
| 人間関係・組織 | 会社の役割分担、友達ネットワーク、口コミの広がり方 |
| 乗り物・旅行 | 電車の乗り換え、カーナビのルート案内、飛行機の座席クラス |
| 天気・自然現象 | 晴れ・曇り・雨の変化、川の流れ、水の循環 |

### Phase 4: 情報を絞り込む

- **「まず理解する3つ」**を決定（これだけわかれば会話に参加できるレベル）
- 全体の中の位置づけを示す（地図を見ている感覚）
- 詳細・発展的内容は後半に配置

### Phase 5: 論調・スタイルを設計

**「知識はあるが押しつけない、信頼できる友人」**の語り口で書く。

- 読者を子供扱いしない。でも専門用語を前置きなしに使わない
- 難しいことを「実は簡単だった」と伝える姿勢で書く
- 「〜なのです」「〜ということです」と丁寧に締める
- 疑問を持ちたくなるような問いかけを随所に入れる
- キャラクター対話なし（ナレーション形式）
- 疑問→回答の構造で進める（見出し疑問文 → 本文で答える形）
- 過度に砕けすぎない・体言止め多用しない・「！」「？」の乱用なし

### Phase 6: HTML生成

1. Tailwind CSS CDN使用（絵文字禁止・Lucide Iconsを使用）
2. 用語解説ボックスを初出時に配置（英語略称は正式名称と日本語訳を記載）
3. コード例は原則不要。必要な場合は「流れ図・概念図」で代替
4. **PDFボタン**を右下に固定配置（`window.print()` で実装。`@media print` でボタン・目次を非表示にし、印刷時に各セクションが途中で切れないよう `break-inside: avoid` を適用）

**セクション構成（この順番で配置すること）**:

| 順 | セクション | 内容 |
|---|-----------|------|
| 1 | ひとことで言えば | 一文サマリー・対象読者・このテーマがカバーする範囲 |
| 2 | なぜ注目されているのか | 歴史的背景・生まれた経緯・私たちの生活との関係 |
| 3 | まず理解する3つ | 会話に参加できるための核心概念を3つに絞る |
| 4 | 主要概念・しくみ | 詳細な概念説明・図（たとえ話を3つ以上含む） |
| 5 | よくある誤解・思い込み | 「実は違った」という気づきポイントと正しい理解 |
| 6 | 料金プラン *(ツール・サービスの場合のみ)* | 無料枠・有料プランの違い・費用感の目安 |
| 7 | 知っておきたい言葉 | 5〜15語、定義付き |
| 8 | もっと知りたい人へ | 関連トピック一覧 |
| 9 | 参考文献・URL | クリッカブルなリンク一覧 |

**「知っておきたい言葉」セクションの構成**:
- このテーマを理解・話すうえで**覚えておくと便利な言葉**を選定してまとめる
- 本文中に登場した用語に限らず、**関連する重要語も含める**
- 各用語は以下の形式で記載する:
  - 用語名（英語略称の場合は正式名称と日本語訳を併記）
  - 一文の定義（中学生でもわかる言葉で）
  - 関連する他の用語へのクロス参照（あれば）
- 用語数の目安: 5〜15語（多すぎると参照性が下がる）

**詳細** → [references/html-structure.md](references/html-structure.md)

### Phase 7: 批判的レビュー（サブエージェント）

**文系の大人が初めてこのテーマを調べた**視点で、忖度なくレビューする。

```
Task({
  subagent_type: "generalPurpose",
  description: "文系の大人視点での図解レビュー",
  readonly: true,
  prompt: `
    あなたは「専門的な理数系の知識は持っていないが、好奇心があり日常の経験は豊富な大人」という立場です。
    今日初めて{トピック名}に興味を持ち、調べ始めました。
    難しい説明や専門用語が前置きなしに出てくると読む気が失せますが、
    丁寧に説明してくれるなら十分理解できます。

    以下のHTML図解を読んで、**忖度なく**批判してください：

    【HTMLファイルパス】{path}

    ## レビュー観点

    1. **わからない言葉**: 専門用語・カタカナ語・英語略語で説明なしに出てくるものはないか
    2. **たとえ話の適切さ**: 日常の経験からイメージできるたとえか。「なるほど」と声が出るくらい具体的か
    3. **情報の過不足**: 「もっと知りたかった」「多すぎて疲れた」はないか
    4. **背景の伝わり方**: なぜこのテーマが生まれたのか・なぜ今重要なのかが伝わるか
    5. **誤解の解消**: よくある思い込みが具体的に取り上げられているか
    6. **料金プランの明確さ** *(ツールの場合)*: 無料枠の制限・有料プランの価格感が分かるか
    7. **学習の連鎖**: 読み終わった後、「次にこれも調べてみたい」と思えたか
    8. **参考文献の質**: URLや参考資料はすぐ確認できるか

    ## 出力形式

    ### 致命的な問題（これがあると読み続けられない）
    - ...

    ### 改善すべき点（直せばもっと伝わる）
    - ...

    ### 良かった点（このまま残すべき）
    - ...

    ### 総評
    5段階評価: ⭐⭐⭐☆☆（3/5）のように
    一言: 「○○だから△△な人には難しい」のように
  `
})
```

### Phase 8: ブラッシュアップ

Phase 7のレビュー結果を踏まえてHTMLを修正する。

**優先度**:
1. 致命的な問題 → 必ず修正
2. 改善すべき点 → 可能な限り対応
3. 良かった点 → 維持

**よくある修正パターン**:
- 用語解説ボックスの追加
- 専門的すぎるたとえ話を、日常の経験に基づくたとえに差し替え
- 参考URLの追加・整理
- 関連トピックの追加

### Phase 9: デプロイ

1. `output/` ディレクトリに `{スラッグ}.html` として保存
2. デプロイ実行:

**Windows（PowerShell）の場合:**

```powershell
npx --yes surge output/{スラッグ}.html --domain curimap-[スラッグ].surge.sh
```

3. 完了報告:

```
完成・公開完了: 【図解タイトル】

（図解の内容を1〜2文で要約）

公開URL:
https://curimap-スラッグ.surge.sh

この図解で理解できること:
- （主要ポイントを3〜5個）

次に調べると面白いトピック:
- （関連トピックを2〜3個）
```

## 品質チェックリスト

### 必須

- [ ] 冒頭に「ひとことで言えば」のサマリーがある
- [ ] **なぜ注目されているのか**セクションがある（歴史・背景・日常との関係）
- [ ] 「まず理解する3つ」のような絞り込みがある
- [ ] テーマ固有の用語には「用語解説ボックス」がある（初出時）
- [ ] 英語略称・専門用語は初出時に日本語の説明が展開されている
- [ ] たとえ話が3つ以上ある（日常の経験だけを使い、かつ正確性を損なわないもの）
- [ ] コード例が必要な場合は「図や流れ図」で代替されている
- [ ] **よくある誤解・思い込み**セクションがある（具体的な気づきポイントと正しい理解）
- [ ] **料金プラン**セクションがある（ツール・サービスの場合のみ）
- [ ] **知っておきたい言葉**（5〜15語、定義付き）が「もっと知りたい人へ」の直前にある
- [ ] **関連トピック**（もっと知りたい人へ）がある
- [ ] **参考文献・URL**がクリッカブルなリンクで末尾にある

### デザイン

- [ ] Lucide iconのみ使用（絵文字禁止）
- [ ] フローチャートまたは概念図が含まれている
- [ ] スマホでも読みやすいレスポンシブ対応
- [ ] 目次がフローティング表示される（デスクトップ）
- [ ] **「PDFで保存」ボタン**が右下に固定表示されている（印刷時は自動非表示）
