# Python 設計墓場・廃止規律記録 (Graveyard of Discarded Rules)

## ■ 本書の役割
本ファイルは、過去に採用されながらも「不完全」「冗長」「非効率」として棄却された設計・規律・ツールを物理的に記録し、**仕様のリバウンド（同じ過ちの反復）を封殺する**ためのものである。

---

## 1. 棄却された構造・ドキュメント

### 1.1 単独の「取扱説明書 (Manual)」
- **棄却理由**: プロジェクト仕様書との内容重複による情報の不一致。
- **墓碑銘**: 「説明書は仕様の鏡であり、物理的に分離してはならない。」

### 1.2 物理的に独立した「requirements.txt」の直接管理
- **棄却理由**: uv による厳格管理への移行。
- **墓碑銘**: 「uv なき環境は、砂上の楼閣である。」

### 1.3 精神としての「産業現場での生存能力」
- **棄却理由**: 開発対象が「産業現場で動くアプリ」そのものではなく、補助ツールへとシフトしたため。
- **墓碑銘**: 「目的が手段を定義する。戦場が変われば、武器も変わる。」

### 1.4 物理的生存能力への執着
- **棄却理由**: PLCとの直接通信や過酷な環境耐性よりも、データの整合性とエンジニアの工数削減こそが真の「生存」であると再定義したため。
- **墓碑銘**: 「1ビットの論理は、1万回の通信リトライより重い。」

---

## 2. 廃止された開発プロセス

### 2.1 AI による「自動的な古いリビジョンの削除」
- **棄却理由**: 人間による最終監査の欠落。
- **墓碑銘**: 「最終判断は常に人間にある。AIはパージの引き金を引いてはならない。」

### 2.2 全ナレッジベースの常時ロード
- **棄却理由**: AIのコンテキスト（短期記憶）の浪費と推論精度の低下。
- **代替案**: Layer 1-3 に階層化された「オンデマンド・ロード」。
- **墓碑銘**: 「過剰な知識は、思考の霧となる。」

---

## 3. 棄却された命名・ディレクトリ規則

### 3.1 `4_取扱説明書.md` 等の連番命名 (プロジェクトルート直下)
- **棄却理由**: 管理の煩雑化。
- **墓碑銘**: 「意味の境界で物理的に隔離せよ。」

### 3.2 UI/Core 内部へのデータ型定義 (Schemas) の混在
- **棄却理由**: 循環参照の誘発とexe化生存率の低下。
- **墓碑銘**: 「中立地帯なき設計に、平安はない。」

### 3.3 プロジェクトごとの「setup_venv.py」による手動環境構築
- **棄却理由**: uv による自動化・決定論的管理への完全移行。手動管理はヒューマンエラーの温床。
- **墓碑銘**: 「自動化できる苦労を、美徳としてはならない。」
