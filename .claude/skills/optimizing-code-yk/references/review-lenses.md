# M1 コードチェック — 手順とレンズ

## 手順

### Step 1: スコープとスタック

[scope.md](scope.md) · [routing.md](routing.md) を Read。リポ · 任意パスを確定。

### Step 2: Web 調査（公式推奨の SSOT）

1. **`researching-web` の `SKILL.md` を Read**
2. スタックに応じたクエリで調査（routing.md の例）
3. 出力を保持: **URL 付き要約**（非推奨 API · 推奨パターン · バージョン注意）

この結果は Step 4 の **全サブエージェント prompt に貼る**。

### Step 3: 事前整理（親）

| やること | 条件 |
|----------|------|
| `ReadLints` / 既知の lint 結果確認 | 対象ファイルが開いている・直近変更あり |
| `git diff` | ユーザーが Shell 明示時のみ |
| 機械的指摘をサブに渡さない | 常時 |

### Step 4: レンズ選定と並列レビュー

1. 下表から **2〜3 レンズ**（デフォルト 3: L-api · L-perf · L-types）
2. **`reviewing-with-subagents` の `SKILL.md` を Read**
3. **同一ターンで** `Task` を 2〜3 本並列起動:
   - `subagent_type`: `explore` · `readonly`: `true`（M1 は原則 Read のみ）
   - 各 prompt に含めるもの:
     - レンズ ID と「見る/見ない」
     - 対象パス一覧
     - Step 2 の Web 調査要約（URL 付き）
     - 出力形式（下記）
     - **他レンズの領域に踏み込まない**

**サブへの出力形式（必須）:**

```markdown
## 採用レンズ
L-api（例）

## この視点の内容
（1〜3 文）

## 所見
| ID | 重大度 | ファイル | 内容 | 根拠（URL または行） |
|----|--------|----------|------|---------------------|
| R1 | MAJOR | path | ... | ... |
```

### Step 5: 統合（親）

1. 所見 ID を **`O1` `O2` …** で通し番号に再採番（サブの R* をマージ）
2. 重複は 1 件にまとめる
3. 対立は根拠の強さで整理（単純マージ禁止）
4. [報告テンプレ](#報告テンプレ) で返す

---

## レンズ一覧

| ID | 見ること | 見ないこと | Web 調査の使い方 |
|----|----------|------------|------------------|
| **L-api** | 非推奨 API · 削除予定 · 公式推奨への乖離 | 命名 · フォーマット | Step 2 の公式 doc を正とする |
| **L-perf** | 明らかな再レンダー · 重い effect · 巨大依存 |  premature optimization | React/Next の性能 tag があれば参照 |
| **L-types** | `any` · unsafe cast · null 境界 | ドメイン仕様の是非 | TypeScript 公式 ·  strict 推奨 |
| **L-a11y** | フォーカス · aria · キーボード（UI 変更時） | 色の好み | `A11Y_RULES` · PLAYWRIGHT a11y 参照 |
| **L-deps** | 古い major · 既知 CVE · 未使用 dep | パッチ版の美学 | package.json + 公式 changelog |

**デフォルト:** L-api + L-perf + L-types  
**UI 中心の依頼:** L-api + L-perf + L-a11y  
**ライブラリ更新疑い:** L-api + L-deps + L-types

---

## 重大度

| 級 | 意味 | M2 での扱い |
|----|------|-------------|
| **BLOCKER** | バグ · 非推奨の即時削除 · セキュリティ | 優先修正候補 |
| **MAJOR** | 公式推奨からの明確な逸脱 · 性能劣化 | ユーザーと相談 or 指定時修正 |
| **MINOR** | 改善余地 · 好み | 報告のみが多い |

---

## 報告テンプレ

```markdown
[最適化レビュー] {repo}（{scope}）

## 1 行サマリー
BLOCKER {n} · MAJOR {m} · MINOR {k} · 参照 URL {u} 件

## Web 調査（要約）
- {URL}: {1行}

## 所見（統合）
| ID | 級 | レンズ | ファイル | 内容 | 根拠 |
|----|-----|--------|----------|------|------|

## 次にできること
- 「最適化を適用して」（O1, O3 など指定可）
- 「コミットして」（M2 後）
- 資料整合（organizing-documents-yk）
```

---

## アンチパターン

- Web 調査なしで「ベストプラクティス」と断言する
- サブ 1 体で多角的と称する
- M1 でコードを書き換える
- 機能追加まで踏み込む（スコープ外）
