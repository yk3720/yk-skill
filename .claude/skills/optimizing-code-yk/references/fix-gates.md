# M2 最適化修正 — ゲートと手順

## 入口ゲート

| 条件 | M2 可 |
|------|-------|
| 直前または同一ターンで M1 所見（`O*` ID）がある | ✅ |
| 「レビューして修正して」「チェックして直して」と明示 | ✅（同一ターンで M1→M2 連続可） |
| 「ここを直して」のみ · レビューなし | ❌ → M1 を提案 |
| 新機能の追加依頼 | ❌ → `creating-*-yk` |

## 手順

### Step 1: 対象所見の確定

- ユーザーが `O3` 等を指定 → そのみ
- 未指定 → **BLOCKER 全件** + **MAJOR** はユーザー確認または 1 問
- **MINOR のみ** → 一括修正しない（明示があれば可）

### Step 2: 実装委譲

[routing.md](routing.md) でファイル種別ごとに子スキルを **Read してから** 修正。

| 触るパス | 委譲先 |
|----------|--------|
| `components/` · Hooks | `creating-react-yk` |
| `app/` · RSC | `creating-nextjs-yk` |
| `lib/flowchart/` · `@xyflow` | `creating-reactflow-yk` |
| `**/*.py` | `creating-pythoncode-yk` |
| `e2e/` | `using-playwright` |

本スキルは **オーケストレーション** — 子スキルの手順に従い最小差分で直す。

### Step 3: 品質ゲート

`QUALITY_GATE_RULES.md` に従う。

| チェック | コマンド例（リポによる） |
|----------|--------------------------|
| 型 | `npm run typecheck` |
| 単体 | `npm run test` |
| E2E | ユーザー明示 or UI 変更大のとき `npm run test:e2e` |
| Python | `npm run excel:test` 等リポ定義 |

**失敗時:** 修正を戻すか、失敗ログを報告して停止。無理に commit しない。

### Step 4: 気づきの蒸留（任意）

非自明なパターンが出たら **M2 完了後**に `distilling-rules-yk` を提案（自動実行しない）。

### Step 5: 報告 · Git

- [報告テンプレ](#報告テンプレ) で返す
- **`git commit` / `git push` はしない**

---

## 修正の境界

| やる | やらない |
|------|----------|
| 所見に対応する最小差分 | 無関係リファクタ |
| 非推奨 API の置換 | API 設計の変更 |
| 公式推奨パターンへの寄せ | 好みのスタイル統一 |
