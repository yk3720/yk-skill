# スキル作成ルール
## Anthropic 公式ドキュメント「Skill Authoring Best Practices」 要約・ルール集 v1

---

## 1. 核心原則

### 1-1. 簡潔さが最優先

スキルはコンテキストウィンドウを他のすべての情報（システムプロンプト・会話履歴・他スキルのメタデータ）と共有する。  
SKILL.md が読み込まれた後は、すべてのトークンが会話履歴と競合する。

> **「Claude はすでに賢い」を前提とせよ。Claude が知っていることは書かない。**

各記述に対して以下を自問する：
- この説明は Claude に本当に必要か？
- Claude はすでにこれを知っているのではないか？
- このパラグラフはトークンコストに見合うか？

**良い例（約50トークン）:**
```python
import pdfplumber
with pdfplumber.open("file.pdf") as pdf:
    text = pdf.pages[0].extract_text()
```

**悪い例（約150トークン）:** PDF とは何か・なぜ pdfplumber を使うか、などの説明を長々と書く → 不要。

---

### 1-2. 自由度を適切に設定する

タスクの性質に応じて、指示の具体度を調整する。

| 自由度 | 使う場面 | 指示形式 |
|--------|----------|----------|
| 高 | 複数のアプローチが有効・文脈依存の判断 | テキストによる方針・手順 |
| 中 | 推奨パターンがある・設定で挙動が変わる | パラメータ付きの疑似コード・スクリプトテンプレート |
| 低 | 操作が脆弱・一貫性が必須・順序固定 | 完全なコマンド・「このまま実行せよ」形式 |

**アナロジー：**  
- 崖のある細い橋 → 低自由度（ガードレール必要）  
- 障害物のない広野 → 高自由度（方向だけ示す）

---

### 1-3. 使用するモデル全てでテストする

スキルの有効性は基盤モデルに依存する。

| モデル | 確認ポイント |
|--------|------------|
| Claude Haiku（高速・低コスト） | ガイダンスは十分か？ |
| Claude Sonnet（バランス型） | 指示は明確で効率的か？ |
| Claude Opus（強力な推論） | 過剰説明になっていないか？ |

## 2. スキルの構造

### 2-1. YAML フロントマター必須フィールド

```yaml
---
name: skill-name-here
description: スキルの説明文
---
```

**`name` の制約：**
- 最大 64 文字
- 小文字・数字・ハイフンのみ
- XML タグ禁止
- 予約語禁止：`anthropic`、`claude`

**`description` の制約：**
- 必須（空不可）
- 最大 1024 文字
- XML タグ禁止
- 何をするか・いつ使うかを両方記述する

---

### 2-2. 命名規則

動名詞形（動詞 + -ing）を推奨：

```
processing-pdfs        ✅
analyzing-spreadsheets ✅
managing-databases     ✅
```

避けるべき命名：

```
helper / utils / tools        ❌ 曖昧すぎ
documents / data / files      ❌ 汎用すぎ
anthropic-helper              ❌ 予約語使用
```

---

### 2-3. description の書き方

- **三人称で書く**（一人称・二人称禁止）
  - ✅ `Processes Excel files and generates reports`
  - ❌ `I can help you process Excel files`
  - ❌ `You can use this to process Excel files`
- 「何をするか」と「いつ使うか」の両方を含める
- 具体的なキーワードを含める（発見精度が上がる）

**良い description の例：**
```yaml
description: Extract text and tables from PDF files, fill forms, merge documents.
  Use when working with PDF files or when the user mentions PDFs, forms, or document extraction.
```

## 3. プログレッシブ開示パターン

### 3-1. SKILL.md は目次として機能させる

- **SKILL.md 本文は 500 行以内**を維持する
- 詳細情報は別ファイルに分離し、必要時のみ読み込ませる
- 参照は **1段階のみ**（SKILL.md → 参照ファイル、それ以上深くしない）

**ディレクトリ構成例：**
```
pdf/
├── SKILL.md          # メイン（トリガー時に読み込み）
├── FORMS.md          # フォーム記入ガイド（必要時のみ）
├── reference.md      # API リファレンス（必要時のみ）
└── scripts/
    ├── analyze_form.py
    └── fill_form.py
```

---

### 3-2. 参照の深さは1層まで

**悪い例（深すぎる）：**
```
SKILL.md → advanced.md → details.md → 実際の情報
```

**良い例（1層のみ）：**
```
SKILL.md → advanced.md（実際の情報がここに）
SKILL.md → reference.md（実際の情報がここに）
```

---

### 3-3. 100行超えのファイルには目次を付ける

```markdown
# API Reference

## Contents
- 認証・セットアップ
- コアメソッド（CRUD）
- 高度な機能
- エラーハンドリング
- コード例
```

## 4. ワークフローとフィードバックループ

### 4-1. 複雑なタスクにはワークフローを使う

複数ステップの処理は明確な連番で分解し、チェックリスト形式で提示する：

```markdown
## タスク進捗
- [ ] Step 1: フォームを解析する
- [ ] Step 2: フィールドマッピングを作成する
- [ ] Step 3: マッピングを検証する
- [ ] Step 4: フォームに記入する
- [ ] Step 5: 出力を確認する
```

---

### 4-2. フィードバックループを実装する

**「検証 → 修正 → 繰り返し」パターン**を明示的に組み込む：

```markdown
1. ドキュメントを編集する
2. 即座に検証する: `python validate.py`
3. 検証失敗時 → エラーを確認して修正 → 再検証
4. 検証通過後のみ次へ進む
```

---

## 5. コンテンツガイドライン

### 5-1. 時間依存情報を書かない

**悪い例：**
```markdown
2025年8月以前は旧 API を使用。それ以降は新 API を使用。
```

**良い例：**
```markdown
## 現在の方法
v2 API エンドポイントを使用: `api.example.com/v2/messages`

## 旧パターン
<details>
<summary>Legacy v1 API（廃止: 2025-08）</summary>
...
</details>
```

---

### 5-2. 用語を統一する

1つの概念には1つの用語を使い続ける：

| ✅ 統一 | ❌ 混在 |
|---------|---------|
| 常に「API エンドポイント」 | endpoint / URL / route / path が混在 |
| 常に「フィールド」 | field / box / element / control が混在 |
| 常に「抽出する」 | extract / pull / get / retrieve が混在 |

---

## 6. よく使うパターン

### 6-1. テンプレートパターン

厳密に出力形式を固定したい場合：
```markdown
**必ずこのテンプレートを使用すること：**
```

柔軟性を持たせたい場合：
```markdown
**基本的にこの形式を使用するが、文脈に応じて調整すること：**
```

---

### 6-2. 入出力例パターン

出力品質が例に依存するタスクでは、具体的な入出力ペアを提示する：

```markdown
**例1:**
入力: ユーザー認証を JWT で実装した
出力:
feat(auth): implement JWT-based authentication
```

---

### 6-3. 条件分岐ワークフローパターン

```markdown
1. 変更の種類を判断する：
   **新規コンテンツを作成？** → 「作成ワークフロー」へ
   **既存コンテンツを編集？** → 「編集ワークフロー」へ
```

## 7. スキル開発のイテレーション

### 7-1. 評価ファーストで開発する

スキルを書く前に、まず評価（テストケース）を作る：

```
1. スキルなしで Claude にタスクを実行させる
2. 失敗点・不足するコンテキストを記録する
3. 3つ以上の評価シナリオを作成する
4. ベースラインを計測する
5. 最小限のスキルを書いてギャップを埋める
6. 評価を繰り返して改善する
```

---

### 7-2. Claude A / Claude B 方式で改善する

| 役割 | 作業 |
|------|------|
| Claude A（設計者） | SKILL.md の作成・改善を手伝う |
| Claude B（実行者） | 実際のタスクでスキルを使ってテストする |

```
Claude B の挙動を観察 → 問題を Claude A に共有 → 改善案を適用 → 再テスト
```

---

### 7-3. Claude がスキルをどう使うかを観察する

- 予期しないファイルの読み方をしている → 構造が直感的でない
- 重要ファイルへの参照をたどれない → リンクが不明確
- 同じファイルを何度も読む → SKILL.md 本体に移すべき内容かも
- 特定ファイルを一切開かない → 不要なファイルか、シグナルが弱い

## 8. スクリプトを含むスキル

### 8-1. エラーを Claude に丸投げしない

**悪い例：**
```python
return open(path).read()  # 失敗したら Claude が何とかする
```

**良い例：**
```python
try:
    with open(path) as f:
        return f.read()
except FileNotFoundError:
    # デフォルト値で続行
    return ""
```

---

### 8-2. マジックナンバーを排除する

```python
# 悪い例
TIMEOUT = 47  # なぜ 47？

# 良い例
# HTTP リクエストは通常 30 秒以内に完了する
# 低速な接続を考慮して余裕を持たせた値
REQUEST_TIMEOUT = 30
```

---

### 8-3. スクリプトの「実行」と「参照」を明示する

```markdown
# 実行させる場合
analyze_form.py を実行してフィールドを抽出してください

# 参照させる場合
抽出アルゴリズムの詳細は analyze_form.py を参照してください
```

---

### 8-4. Windows スタイルのパスを使わない

```
✅ scripts/helper.py
✅ reference/guide.md
❌ scripts\helper.py
❌ reference\guide.md
```

---

## 9. アンチパターン（やってはいけないこと）

| アンチパターン | 理由 |
|--------------|------|
| 選択肢を多く提示しすぎる | 判断を迷わせる。デフォルトを示して「代替」として補足する |
| 時間依存の条件分岐を書く | スキルが古くなった時に誤動作する |
| 参照を 2 段階以上ネストする | Claude が途中で止まり情報を取りきれない可能性がある |
| Windows スタイルのパスを使う | Unix 環境でエラーになる |
| Claude がすでに知っていることを説明する | トークンの無駄 |
| ツール名をサーバー名なしで書く | 「tool not found」エラーになる（例：`BigQuery:bigquery_schema` と書く） |

## 10. リリース前チェックリスト

### コア品質
- [ ] description に「何をするか」と「いつ使うか」が両方含まれる
- [ ] description にキーワードが含まれる
- [ ] SKILL.md 本文が 500 行以内
- [ ] 詳細情報は別ファイルに分離済み（必要な場合）
- [ ] 時間依存情報なし（あれば「旧パターン」セクションへ）
- [ ] 全体で用語が統一されている
- [ ] 例が具体的（抽象的でない）
- [ ] ファイル参照が 1 段階のみ
- [ ] ワークフローのステップが明確

### スクリプト・コード
- [ ] スクリプトがエラーを Claude に丸投げしていない
- [ ] マジックナンバーにコメントで根拠を記述している
- [ ] 必要なパッケージが明記されている
- [ ] 実行/参照の意図が明示されている
- [ ] Windows スタイルのパスを使っていない
- [ ] 重要な操作に検証・フィードバックループがある

### テスト
- [ ] 評価シナリオが 3 つ以上ある
- [ ] Haiku・Sonnet・Opus でテスト済み
- [ ] 実際のユースケースでテスト済み
