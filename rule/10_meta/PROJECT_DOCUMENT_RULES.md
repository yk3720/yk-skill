# プロジェクトドキュメント体系規則
## Project Document Taxonomy — YK 共通 v1

**用途:** 個人アプリの **企画フォルダ内ドキュメント種別 · フォルダ構成 · 正本の置き場** の SSOT。  
**ステータス:** active（L1 · 2026-06-23 グリル確定）  
**関連:** `10_meta/APP_PROJECT_RULES.md`（ライフサイクル · handoffs · 再開）· `RULE_INDEX.md` No 25  
**実例（移行前）:** [flowchart-web 企画](c:/yk-memo/00.ai-driven-school/個人テーマ_フローチャートアプリ/) — 旧 `01_product/` 等

**最終更新:** 2026-06-23

---

## 1. 解く問題

| 症状 | 原因 |
|------|------|
| 「メモ」と「仕様」の区別が曖昧 | フォルダ名が作業種別ではなく作業段階のみ |
| ADR · decision-log · AGENTS が散在 | 旧 4 フォルダ構成と憲法の置き場が未統一 |
| 2 本目以降のツール立ち上げが毎回バラバラ | 正式な種別定義が rule にない |
| 仕様とコードのズレ判断が毎回口頭 | 正本の優先順位が文書化されていない |

**目標:** 6 種の正式ドキュメント種別で企画フォルダを揃え、**複数人 · 別ツール向けのベース**を作る。セッション運用（handoffs）は `APP_PROJECT_RULES` のまま分離。

---

## 2. 六種ドキュメント（MUST）

番号 + 日本語短縮のフォルダ名を使う。`@` 指定と可読性のバランスを優先。

| No | フォルダ | 種別 | 載せる内容 | 載せない内容 |
|----|----------|------|------------|--------------|
| 01 | `01_要求定義/` | 要求定義 | プロダクト要求 · MVP · 完成定義 · ユーザーストーリー · 調査（決定前） | 技術詳細 · ADR 本文 · セッション進捗 |
| 02 | `02_機能設計/` | 機能設計 | 画面構成 · ユースケース · フロー · フェーズ計画 | 列定義 · DB スキーマ · 実装手順 |
| 03 | `03_技術仕様/` | 技術仕様 | データモデル · API · 列の意味 · 技術方針 · **ADR** | 経緯タイムライン · セッション記録 |
| 04 | `04_リポジトリ構造/` | リポジトリ構造 | コード置き場 · パスマップ · 環境 · デプロイ構成 | ドメイン用語集 · プロダクト要求 |
| 05 | `05_開発ガイドライン/` | 開発ガイドライン | **エージェント憲法** · コーディング方針 · **decision-log**（経緯索引） | 仕様全文 · handoffs 本文 |
| 06 | `06_ユビキタス言語/` | ユビキタス言語 | 用語定義 · 略語 · ドメイン辞書 | 画面ワイヤー · 実装コード |

**MUST NOT:** 上記 6 種を「メモ」「資料」など非公式ラベルで代替しない。

---

## 3. 標準フォルダ構成

```text
{企画フォルダ}/
├── README.md                          # 唯一の索引
├── 新チャット依頼.md                   # 薄いラッパ（ルート固定 · 任意）
├── 00_テーマ/                          # 別枠 — 講座テーマ · 選定表（§5）
├── 01_要求定義/
├── 02_機能設計/
├── 03_技術仕様/
├── 04_リポジトリ構造/
├── 05_開発ガイドライン/
│   └── エージェント憲法.md             # 旧 AGENTS.md の正本（§4）
├── 06_ユビキタス言語/
└── 99_アーカイブ/                      # 別枠 — 履歴 · 立ち上げ期調査
    ├── research/
    └── evidence/
```

**handoffs**（`yk-memo/handoffs/{slug}/`）は企画フォルダ外の **別枠** — セッション SSOT。本 rule の 6 種には含めない。

## 4. エージェント憲法（旧 AGENTS.md）

| 項目 | 決定 |
|------|------|
| 正本 | `05_開発ガイドライン/エージェント憲法.md` |
| ルート `AGENTS.md` | **廃止**（6 種体系との一貫性） |
| 再開時の読む順序 | handoffs HANDOFF → 最新 §4 → **エージェント憲法**（3 ファイル） |

**移行中の例外:** 物理移行が未完了のプロジェクト（例: flowchart-web）は、完了まで現行ルート `AGENTS.md` を参照してよい。移行完了後は stub または削除。

憲法の必須項目 → `APP_PROJECT_RULES.md` §5（内容は本ファイルで再定義しない）。

---

## 5. 別枠（6 種に含めない）

| 置き場 | 役割 | 備考 |
|--------|------|------|
| `handoffs/{slug}/` | セッション · §4 = 次の 1 件 | `APP_PROJECT_RULES` · `handoff-session-work` |
| `00_テーマ/` | 講座テーマ · 選定表 · 報告書（将来） | 全プロジェクト必須かは **未決**（§9） |
| `99_アーカイブ/` | 完了調査 · 手動確認スナップショット · 旧構成の退避 | 再開時は通常 Read しない |

## 6. 旧構成からの吸収ルール

2026-05-24 頃の番号付き 4 フォルダ + ルート憲法を、6 種へ寄せる。

| 旧パス | 新パス | 備考 |
|--------|--------|------|
| `01_product/` | `01_要求定義/` | grill-me · 調査 · MVP · 完成チェックリスト |
| `02_spec/` | `03_技術仕様/` | データモデル · 列定義 · 技術方針 |
| `03_plan/` | `02_機能設計/` | フェーズ計画 · 画面フロー |
| `04_decisions/意思決定記録(ADR)*` | `03_技術仕様/` | Accepted 技術・プロダクト決定 |
| `04_decisions/decision-log.md` | `05_開発ガイドライン/` | 経緯索引（1 行要約 + リンク） |
| `AGENTS.md`（ルート） | `05_開発ガイドライン/エージェント憲法.md` | 内容移行後にルートは廃止 |
| `99_archive/` | `99_アーカイブ/` | 表記統一（アンダースコア） |
| `00_theme/` | `00_テーマ/` | 表記統一 |

**grill-me 昇格（MUST）** — `APP_PROJECT_RULES` §13 と同じ:

1. grill-me で **Accepted** → ADR（`03_技術仕様/`）+ decision-log 1 行  
2. grill-me で **未決** → grill-me §4 のみ（ADR に書かない）  
3. セッション終了 → decision-log 1 行 · handoff §1 要約（ADR 全文コピー禁止）

---

## 7. 新規プロジェクトの最小構成

立ち上げ負荷を抑えるため、**最初は 3 種のみ** 必須とする。

| 必須 | 内容 |
|------|------|
| `01_要求定義/` | プロジェクト概要 または MVP 定義（1 ファイル以上） |
| `04_リポジトリ構造/` | コード置き場 · slug 対応 1 行 |
| `05_開発ガイドライン/エージェント憲法.md` | 境界 · SSOT マップ · 再開 3 ファイル |

**任意（規模に応じて追加）:** `02_機能設計/` · `03_技術仕様/` · `06_ユビキタス言語/` · `99_アーカイブ/`

**追加トリガー（`02` `03` `06` をいつ必須にするか）:** **未決**（§9）。現時点では機能・技術の議論が始まったタイミングで追加を推奨。

## 8. 成熟プロジェクトの移行

既存ファイルが多いプロジェクト（例: flowchart-web）は **6 フォルダすべて** に既存ファイルを振り分ける一括移行とする。

| 順序 | 作業 |
|------|------|
| 1 | 本 rule（No 25）確定 — **先に rule** |
| 2 | 企画フォルダで 6 種 + 別枠フォルダを作成 |
| 3 | 旧パスから §6 の吸収表どおりに移動 |
| 4 | `README.md` 索引更新 · handoffs / エージェント憲法のパス更新 |
| 5 | ルート `AGENTS.md` を stub 化または削除 |

**MUST NOT:** rule 未確定のまま物理移行だけ先行する。

---

## 9. 技術仕様とコードの関係（方針 B）

| 状況 | 扱い |
|------|------|
| 仕様変更 | **仕様書を先に更新**してからコードを合わせる |
| コードと仕様のズレ発見 | 意図確認のうえ、**基本はコードを仕様に合わせる** |
| 意図的なコード先行 | decision-log または ADR に理由を 1 行記録してから仕様を追記 |

**却下した案（参考）:** spec-as-source（仕様のみ編集）· `types.ts` を正本とするコードファースト · 仕様+現状の 2 ファイル併存。

|------|----------|
| 企画フォルダ新設 · 構成整理 | 本ファイル → `APP_PROJECT_RULES` §4 チェックリスト |
