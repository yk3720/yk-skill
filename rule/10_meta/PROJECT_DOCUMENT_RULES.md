# プロジェクトドキュメント体系規則
## Project Document Taxonomy — YK 共通 v1

**用途:** 個人アプリの **企画フォルダ内ドキュメント種別 · フォルダ構成 · 正本の置き場** の SSOT。  
**ステータス:** active（L1 · 2026-06-23 グリル確定）  
**関連:** `10_meta/APP_PROJECT_RULES.md`（ライフサイクル · handoffs · 再開）· `RULE_INDEX.md` No 25  
**実例（移行前）:** [flowchart-web 企画](c:/yk-memo/00.ai-driven-school/個人テーマ_フローチャートアプリ/) — 旧 `01_product/` 等

**最終更新:** 2026-06-23（§11–§12 残グリル確定）

---

## 1. 解く問題

| 症状 | 原因 |
|------|------|
| 「メモ」と「仕様」の区別が曖昧 | フォルダ名が作業種別ではなく作業段階のみ |
| ADR · decision-log · AGENTS が散在 | 旧 4 フォルダ構成と憲法の置き場が未統一 |
| 2 本目以降のツール立ち上げが毎回バラバラ | 正式な種別定義が rule にない |
| 仕様とコードのズレ判断が毎回口頭 | 正本の優先順位が文書化されていない |

**目標:** 6 種の正式ドキュメント種別で企画フォルダを揃え、**複数人 · 別ツール向けのベース**を作る。セッション運用（handoffs）は `APP_PROJECT_RULES` のまま分離。

---

## 2. 六種ドキュメント（MUST）

番号 + 日本語短縮のフォルダ名を使う。`@` 指定と可読性のバランスを優先。

| No | フォルダ | 種別 | 載せる内容 | 載せない内容 |
|----|----------|------|------------|--------------|
| 01 | `01_要求定義/` | 要求定義 | プロダクト要求 · MVP · 完成定義 · ユーザーストーリー · 調査（決定前） | 技術詳細 · ADR 本文 · セッション進捗 |
| 02 | `02_機能設計/` | 機能設計 | 画面構成 · ユースケース · フロー · フェーズ計画 | 列定義 · DB スキーマ · 実装手順 |
| 03 | `03_技術仕様/` | 技術仕様 | データモデル · API · 列の意味 · 技術方針 · **ADR** | 経緯タイムライン · セッション記録 |
| 04 | `04_リポジトリ構造/` | リポジトリ構造 | コード置き場 · パスマップ · 環境 · デプロイ構成 | ドメイン用語集 · プロダクト要求 |
| 05 | `05_開発ガイドライン/` | 開発ガイドライン | **エージェント憲法** · コーディング方針 · **decision-log**（経緯索引） | 仕様全文 · handoffs 本文 |
| 06 | `06_ユビキタス言語/` | ユビキタス言語 | 用語定義 · 略語 · ドメイン辞書 | 画面ワイヤー · 実装コード |

**MUST NOT:** 上記 6 種を「メモ」「資料」など非公式ラベルで代替しない。

## 3. 標準フォルダ構成

```text
{企画フォルダ}/
├── README.md                          # 唯一の索引
├── 新チャット依頼.md                   # 薄いラッパ（ルート固定 · 任意）
├── 00_テーマ/                          # 別枠 — 講座テーマ · 選定表（§5）
├── 01_要求定義/
├── 02_機能設計/
├── 03_技術仕様/
├── 04_リポジトリ構造/
├── 05_開発ガイドライン/
│   └── エージェント憲法.md             # 旧 AGENTS.md の正本（§4）
├── 06_ユビキタス言語/
└── 99_アーカイブ/                      # 別枠 — 履歴 · 立ち上げ期調査
    ├── research/
    └── evidence/
```

**handoffs**（`yk-memo/handoffs/{slug}/`）は企画フォルダ外の **別枠** — セッション SSOT。本 rule の 6 種には含めない。

## 4. エージェント憲法（旧 AGENTS.md）

| 項目 | 決定 |
|------|------|
| 正本 | `05_開発ガイドライン/エージェント憲法.md` |
| ルート `AGENTS.md` | **廃止**（6 種体系との一貫性） |
| 再開時の読む順序 | handoffs HANDOFF → 最新 §4 → **エージェント憲法**（3 ファイル） |

**移行中の例外:** 物理移行が未完了のプロジェクト（例: flowchart-web）は、完了まで現行ルート `AGENTS.md` を参照してよい。移行完了後は stub または削除。

憲法の必須項目 → `APP_PROJECT_RULES.md` §5（内容は本ファイルで再定義しない）。

## 6. 旧構成からの吸収ルール

2026-05-24 頃の番号付き 4 フォルダ + ルート憲法を、6 種へ寄せる。

| 旧パス | 新パス | 備考 |
|--------|--------|------|
| `01_product/` | `01_要求定義/` | grill-me · 調査 · MVP · 完成チェックリスト |
| `02_spec/` | `03_技術仕様/` | データモデル · 列定義 · 技術方針 |
| `03_plan/` | `02_機能設計/` | フェーズ計画 · 画面フロー |
| `04_decisions/意思決定記録(ADR)*` | `03_技術仕様/` | Accepted 技術・プロダクト決定 |
| `04_decisions/decision-log.md` | `05_開発ガイドライン/` | 経緯索引（1 行要約 + リンク） |
| `AGENTS.md`（ルート） | `05_開発ガイドライン/エージェント憲法.md` | 内容移行後にルートは廃止 |
| `99_archive/` | `99_アーカイブ/` | 表記統一（アンダースコア） |
| `00_theme/` | `00_テーマ/` | 表記統一 |

**grill-me 昇格（MUST）** — `APP_PROJECT_RULES` §13 と同じ:

1. grill-me で **Accepted** → ADR（`03_技術仕様/`）+ decision-log 1 行  
2. grill-me で **未決** → grill-me §4 のみ（ADR に書かない）  
3. セッション終了 → decision-log 1 行 · handoff §1 要約（ADR 全文コピー禁止）

## 7. 新規プロジェクトの最小構成

立ち上げ負荷を抑えるため、**最初は 3 種のみ** 必須とする。

| 必須 | 内容 |
|------|------|
| `01_要求定義/` | プロジェクト概要 または MVP 定義（1 ファイル以上） |
| `04_リポジトリ構造/` | コード置き場 · slug 対応 1 行 |
| `05_開発ガイドライン/エージェント憲法.md` | 境界 · SSOT マップ · 再開 3 ファイル |

**任意（規模に応じて追加）:** `02_機能設計/` · `03_技術仕様/` · `06_ユビキタス言語/` · `99_アーカイブ/`

**追加トリガー:** §12.1（`02` `03` `06` をいつ必須にするか）。

## 8. 成熟プロジェクトの移行

既存ファイルが多いプロジェクト（例: flowchart-web）は **6 フォルダすべて** に既存ファイルを振り分ける一括移行とする。

| 順序 | 作業 |
|------|------|
| 1 | 本 rule（No 25）確定 — **先に rule** |
| 2 | 企画フォルダで 6 種 + 別枠フォルダを作成 |
| 3 | 旧パスから §6 の吸収表どおりに移動 |
| 4 | `README.md` 索引更新 · handoffs / エージェント憲法のパス更新 |
| 5 | ルート `AGENTS.md` を stub 化または削除 |

**MUST NOT:** rule 未確定のまま物理移行だけ先行する。

## 9. 技術仕様とコードの関係（方針 B）

| 状況 | 扱い |
|------|------|
| 仕様変更 | **仕様書を先に更新**してからコードを合わせる |
| コードと仕様のズレ発見 | 意図確認のうえ、**基本はコードを仕様に合わせる** |
| 意図的なコード先行 | decision-log または ADR に理由を 1 行記録してから仕様を追記 |

**却下した案（参考）:** spec-as-source（仕様のみ編集）· `types.ts` を正本とするコードファースト · 仕様+現状の 2 ファイル併存。

## 10. SDD 取り込み（Spec-Driven Development）

**背景:** GitHub Spec Kit 等の SDD 思想を、YK の **三層モデル + 6 種 + handoffs** にマッピングする。経緯・却下理由の詳細 → [`yk-memo/research/調査_スペック駆動開発.md` §11](c:/yk-memo/research/調査_スペック駆動開発.md)。

| 論点 | 決定 |
|------|------|
| Spec Kit 丸ごと導入 | **却下** — handoffs §4 と二重 SSOT |
| JSON Schema で仕様検証 | **却下（当面）** — Markdown AC + fixture + E2E で足りる |
| 仕様↔コード | **方針 B**（§9）— spec-anchored 寄り |

### 10.1 Spec Kit 工程 → YK 正本

| 工程 | YK 正本 | DoD |
|------|---------|-----|
| Constitution | `05_開発ガイドライン/エージェント憲法.md` | `APP_PROJECT_RULES` §5 必須項目 |
| Specify | `01_要求定義/` | **AC 表**（§10.2） |
| Plan | `02_機能設計/` + `03_技術仕様/` + ADR | フェーズ計画 · Accepted ADR |
| Tasks | handoffs 最新 **§4** | チェックリスト（`AC-N` 番号は使わない） |
| Implement | `yk-tool/` 等 | Vitest → Playwright（`PLAYWRIGHT_RULES`） |

**MUST NOT:** Spec Kit の `specs/` · `.specify/` を並行 SSOT として追加する。

### 10.2 受け入れ条件（AC）3 層

| 層 | 記号 | 置き場 | 用途 |
|----|------|--------|------|
| マイルストーン AC | `AC-1` … `AC-N` | `01_要求定義/`（例: `MVP定義.md`） | MVP · M2 等のフェーズ完了 |
| 技術ゲート | `G-1` 等（任意） | `03_技術仕様/` または ADR 1 節 | DB · API 等 |
| セッション DoD | チェックボックス | handoffs セッション MD **§4** | 今回 1 件の実行完了 |

**マイルストーン AC 表（標準）:** 列は `#` · `条件` · `確認方法`（E2E spec · fixture · 手動のいずれか 1 行）。実例 → [flowchart `MVP定義.md`](c:/yk-memo/00.ai-driven-school/個人テーマ_フローチャートアプリ/01_要求定義/MVP定義.md)。

**MUST:** `AC-N` 番号は **マイルストーン層のみ**。§4 に `AC-3 を満たす` と書かない。  
**MUST:** MVP+（完了条件外）は AC 表の別表に「完了条件外」と明記。

完成定義の分離（4 段階）→ `APP_PROJECT_RULES` §8。

### 10.3 handoffs §4（Tasks 相当）

| 段階 | 正本 | §4 に書くか |
|------|------|-------------|
| 完成（A〜D ✅） | `完成チェックリスト.md` | 書かない |
| MVP / M2 / M3 | `MVP定義.md` 等 | 書かない（リンクのみ可） |
| **§4 Session** | 最新セッション MD §4 | **ここだけ** |
| 検証 | §4 内 1 行 · `PLAYWRIGHT_RULES` | 必要時のみ |

**§4 task packet（必須 3 ブロック）:** 完了の定義 · やる/やらない · 依頼文 — 追加見出しは設けない。

**§4 に載せない:** 人間専用タスク（→ HANDOFF §6）· ロードマップ全体 · PRD/ADR/rule 全文（パスリンクのみ）。

**OpenAPI / BDD:** §12.3 · §12.4。

## 11. ファイル命名規約（2026-06-23 確定）

**方針:** 正式名は **少数の canonical ファイル** に固定し、それ以外は **接頭辞パターン** で揃える。`プロダクト要求定義書.md` のような重い固定名は **採用しない**。

### 11.1 フォルダ別

| フォルダ | 固定名（推奨 · 新規 MUST） | 接頭辞パターン（追加ファイル） |
|----------|---------------------------|-------------------------------|
| ルート | `README.md` · `新チャット依頼.md`（任意） | — |
| `01_要求定義/` | `プロジェクト概要.md` · `MVP定義.md` **または** `完成チェックリスト.md` | `調査_{テーマ}.md` · `grill-me_{YYYY-MM-DD}_{論題}.md` · `計画_{YYYY-MM-DD}_{論題}.md` |
| `02_機能設計/` | `フェーズ計画.md`（複数フェーズ時） | `画面構成_{名前}.md` · `シナリオ_{名前}.md`（BDD 任意 · §12.4） |
| `03_技術仕様/` | `データモデル.md` · `技術方針.md` · `意思決定記録(ADR).md` | ドメイン名（例: `フローチャート記述ルール.md`）· `openapi.yaml`（§12.3） |
| `04_リポジトリ構造/` | `リポジトリ構造.md` | — |
| `05_開発ガイドライン/` | `エージェント憲法.md` · `decision-log.md` | — |
| `06_ユビキタス言語/` | `README.md`（索引 · 任意） | `用語_{ドメイン}.md` |
| `00_テーマ/` | `テーマ.md` · `選定表_記入稿.md`（講座時） | `報告書_{YYYY}.md`（将来） |
| `99_アーカイブ/` | `README.md` | `research/` · `evidence/` 配下は日付入り可 |

**レガシー:** `相談_{YYYY-MM-DD}_*.md` は flowchart 等に残存可。新規は `grill-me_` を推奨（`APP_PROJECT_RULES` §13）。

**MUST NOT:** 同一役割を `概要.md` と `プロジェクト概要.md` のように **別名で二重作成**する。

実例 → [flowchart-web 企画](c:/yk-memo/00.ai-driven-school/個人テーマ_フローチャートアプリ/) §13。

## 12. フォルダ追加トリガー（2026-06-23 確定）

### 12.1 `02` `03` `06` の必須化

| フォルダ | いつ必須にするか | それまでは |
|----------|------------------|------------|
| **`02_機能設計/`** | 画面 2 以上 · ユーザーフロー · 業務シナリオ · フェーズ 2 以降を **初めて**設計するとき | `01_要求定義/MVP定義.md` のみで可 |
| **`03_技術仕様/`** | データモデル · 永続化 · 外部 API · 列定義 · **ADR を初めて**書くとき | 技術メモを `01_要求定義/調査_*` に一時置き可（昇格後に移動） |
| **`06_ユビキタス言語/`** | **当面なし（任意のまま）** | 用語は `03_技術仕様/`（例: `列の意味.md`）で足りる |

**`06` を追加する任意トリガー:** ドメイン用語が **10 語超** · 非エンジニア読者が増えた · `03` が肥大化したとき。

### 12.2 `00_テーマ/`

| 状況 | 扱い |
|------|------|
| **講座提出・年度テーマ**に紐づく（例: `00.ai-driven-school/` 配下） | `00_テーマ/` **必須** — `テーマ.md` + 選定表 |
| 個人ツール · 講座外 | **任意** — 作らなくてよい |

### 12.3 OpenAPI 導入トリガー

| 状況 | 扱い |
|------|------|
| **外部システム**が HTTP API を契約として呼ぶ（別クライアント · 社内連携 · 第三者） | `03_技術仕様/openapi.yaml`（または実装リポ内）+ ADR 1 行 |
| Next.js Route Handler を **同一アプリ内のみ**で使用 | OpenAPI **不要** |
| ブラウザ → Supabase 直接 | OpenAPI **不要**（RLS · ADR で足りる） |

### 12.4 BDD（Given-When-Then）

| 状況 | 扱い |
|------|------|
| 全機能に BDD 必須 | **却下** |
| 分岐 **3 以上**の業務フロー · 関係者レビューが必要 | `02_機能設計/シナリオ_{名前}.md` を **任意で**追加（GWT 推奨 · 固定テンプレ不要） |
| MVP · CRUD · 単一画面 | BDD **不要** — AC 表 + E2E で足りる |
